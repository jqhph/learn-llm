# Lesson 5: 混合检索与查询转换

The learner can now implement hybrid retrieval (vector + BM25 with RRF fusion) and three query transformation strategies (Query Rewrite, Multi-Query, HyDE), and A/B test them against the golden evaluation dataset from Lesson 4.

**New capabilities:**
- Implement `BM25Retriever` with `rank_bm25` + `jieba` Chinese tokenization, including `build_index()`, `retrieve()`, `save_index()`, and parameter tuning (k1, b)
- Implement `HybridRetriever` with RRF (Reciprocal Rank Fusion) that fuses vector and BM25 results by rank position, with adjustable k parameter and per-retriever weights
- Implement three `QueryProcessor` strategies: Multi-Query expansion (3-5 variants → separate retrieval → RRF fusion), Query Rewrite (LLM-based conversion from colloquial to formal), HyDE (Hypothetical Document Embedding)
- Build a unified adapter layer (`retrieval_adapter.py`) where all retriever variants share the `Callable[[str], list[dict]]` signature, making them trivially interchangeable with the Lesson 4 evaluator
- Run a four-way comparison experiment (vector-only, BM25-only, hybrid, hybrid+rewrite) and interpret the results
- Understand the production hybrid retrieval pipeline: Query Rewrite → Dual-Recall → RRF → Cross-Encoder → LLM

**Key insight:** Vector and keyword retrieval are complementary, not competing. Vector retrieval excels at semantic matching (synonyms, paraphrases, general intent), while BM25 excels at exact term matching (SKUs, order numbers, product codes, invoice IDs). RRF elegantly fuses their results without requiring score normalization — it only needs rank positions. Query Transformation is the "low-hanging fruit" before retrieval: Query Rewrite should be always-on as it has the lowest cost and most consistent benefit.

**Why:** Pure vector retrieval has a known blind spot for exact-match queries containing specific identifiers (order numbers, SKUs, model numbers) — embedding tokenization breaks these into subwords and loses the exact match signal. In enterprise customer-service Q&A, such queries are common (users asking "where is my order SK20240615"). Hybrid retrieval fixes this blind spot with minimal additional infrastructure. Query transformation additionally fixes the "short query problem" where embedding a short query produces a low-quality vector.

**How to apply:** The learner should now always run hybrid retrieval (vector + BM25) as their default, with Query Rewrite enabled. Next lesson should cover answer quality evaluation (Faithfulness, Answer Relevance, Context Precision) and the RAGAS framework — the learner now has good retrieval, needs to verify that good retrieval leads to good answers.

**Skill gaps observed:**
- No BM25 performance benchmarking against different chunk sizes/strategies — the learner hasn't experimented with how chunking affects keyword retrieval differently from vector retrieval
- The `jieba` tokenizer may miss domain-specific terms; no custom dictionary maintenance has been discussed
- No production-grade BM25 solution (Elasticsearch, Meilisearch) — the current `rank_bm25` is in-memory and doesn't scale to large document collections
- Query transformation introduces additional LLM latency that hasn't been measured or optimized (caching, async, fallback strategies)
- RRF weight tuning is heuristic; no systematic weight search has been performed
