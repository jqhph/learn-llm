# Lesson 4: 检索质量评估与重排序

The learner can now quantify retrieval quality using Recall@K and MRR metrics, build a human-annotated golden test set, and implement a two-stage retrieve-rerank pipeline using Cross-Encoder models. They have an `evaluator.py` module and a `reranker.py` module that integrate with their existing RAG pipeline from Lesson 2-3.

**Key insight:** Evaluation is the prerequisite for optimization — without quantified metrics, every parameter change is guesswork. A 10-query golden dataset is enough to catch 90% of regression. The two-stage paradigm (bi-encoder for coarse retrieval, cross-encoder for fine re-ranking) is the single highest-leverage optimization for improving answer quality in a customer-service RAG system.

**New capabilities:**
- Build a `golden.json` test set with query-chunk relevance annotations (10-20 queries covering easy/medium/hard cases)
- Implement `evaluator.py` with Recall@K (K=1,3,5) and MRR computation
- Run baseline evaluation and generate a structured report with worst-query analysis
- Implement a Cross-Encoder re-ranker using `sentence-transformers` (MiniLM-L-6-v2 or larger)
- Integrate re-ranking into the existing RAG pipeline without changing the retrieval or generation code
- Run A/B comparison experiments (baseline vs re-ranked) on the same golden dataset
- Diagnose retrieval failure modes (synonym gap, cross-chunk answers, keyword coverage, granularity mismatch)

**Skill gaps observed:**
- The learner hasn't yet explored hybrid search (vector + BM25 keyword fusion) — the next logical step for handling SKU/order-number/part-number queries that vector-only retrieval handles poorly.
- No embedding model comparison yet — they're still on text-embedding-3-small vs alternatives like BGE, Cohere, Jina.
- No query rewriting/transformation (e.g., Hypothetical Document Embeddings, query expansion) before retrieval.
- Re-ranking latency on CPU hasn't been measured — the MiniLM-L-6 model is fast enough for demo but may need quantization for production.
- No automated regression testing in CI — golden dataset is run manually, not as part of a pipeline.

**Why:** Without quantified metrics, RAG optimization is subjective and unrepeatable. The learner now has the tooling to make data-driven decisions: every parameter change (chunk_size, embedding model, overlap, top-K, re-ranking on/off) can be evaluated against the same golden dataset, producing comparable scores. This transforms RAG from a "craft" into an engineering discipline.

**How to apply:** The next lesson should introduce hybrid retrieval (vector + BM25) and query transformation techniques. This is the natural next step because: (1) the evaluator now exists to measure improvement; (2) the learner will encounter queries about specific terms/IDs that vector-only retrieval misses; (3) hybrid search is the industry-standard solution for production RAG.
