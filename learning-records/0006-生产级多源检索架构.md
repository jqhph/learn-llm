# 第 6 课：生产级多源检索架构

## 核心发现
课程 0005 中使用"订单号 SK20240615"作为 BM25 精确匹配的例子在**教学层面正确**（展示 BM25 机制），但在**架构层面容易产生误导**——现实中订单数据存储在 MySQL/PostgreSQL 中，可能有上亿条记录，不应放入向量库或 BM25 索引。

## 核心原则：数据分水岭
- **知识型/参考型数据**（政策文档、产品手册、FAQ）→ 向量库（Chroma/Pinecone/Qdrant），量级通常在千~百万条
- **事务型/记录型数据**（订单、物流、库存、用户）→ 留在源数据库，通过 API / Function Calling / SQL 查询

## 生产级四种架构模式
1. **查询路由 (Query Routing)**：先用 LLM/规则分类意图，再路由到不同数据源处理管道
2. **Function Calling (Tool Use)**：给 LLM 查询数据的能力而非数据本身，工业级标准方案
3. **NL2SQL**：LLM 生成 SQL 直查数据库，灵活但不安全风险高，需只读用户+参数化查询+LIMIT+超时
4. **联邦检索 (Federated Retrieval)**：复杂查询分解→多源并行取数→结果融合，适用于 4+ 数据源场景

## 关键实践
- 正则提取订单号/SKU → LLM 兜底（先正则后 LLM，延迟从 200ms 降到 1ms）
- 向量库只占整体查询量的约 30%，订单/物流等结构化查询占 50%+
- NL2SQL 必须使用只读副本（read replica），绝不允许写操作

## 推荐演进路径
路由分类 → Function Calling → (按需引入 NL2SQL) → 联邦检索

**Why:** 理解了为什么课程 0005 的例子不能直接套用到生产环境，以及真实企业级 RAG 系统中各种数据源的正确接入方式。
**How to apply:** 在客服 RAG 系统的下一阶段设计中，用查询路由作为第一道门，将订单/知识库/产品查询分别路由到不同的处理管道。
