第一部分：认识可观测性三层体系

在动手之前，先理解一个重要的架构思想——可观测性的三个层次。
一个 RAG 系统的请求路径很长（用户 → Nginx → FastAPI → LLM API → 向量库 → ...），每个环节都可能是瓶颈。你需要在三个层次同时看到发生了什么，才能快速定位问题。

1.2 可观测性的三大支柱

可观测性领域有三个核心数据类型，称为"三大支柱"：

支柱是什么用途工具
📈 指标 (Metrics)数值化的聚合数据，如请求数、延迟、错误率趋势分析、告警触发、容量规划Prometheus
📝 日志 (Logs)离散的事件记录，每条带时间戳和上下文问题排查、详细调试、审计追踪Loki / ELK
🔗 追踪 (Traces)一次请求经过多个服务的完整路径分布式链路分析、瓶颈定位Langfuse / Jaeger

三大支柱必须协同工作

指标告诉你"有什么问题"，日志告诉你"具体发生了什么"，追踪告诉你"是哪个环节出的问题"。

举个例子：

  1. 指标告警：P95 延迟从 2s 飙到了 10s → 触发告警
  2. 日志查看：搜索这个时间段的日志 → 发现大量 "LLM API timeout" 错误
  3. 追踪下钻：追某次请求的完整链路 → 定位到 OpenAI API 调用阶段消耗了 15s

三个步骤配合，5 分钟内就能定位到"OpenAI API 超时"这个根因。缺了任何一个，都要花几倍的时间。

第二部分：应用层监控 —— 为 RAG API 添加 Prometheus 指标

监控从应用层开始——因为这是最贴近业务的层次。我们先给 RAG API 加上自定义指标。

2.1 什么是 Prometheus 指标？

Prometheus 是一个开源的监控系统与时间序列数据库。它的工作模式是：

  1. 你在代码中暴露一个 /metrics HTTP 端点
  2. Prometheus 服务定期（默认 15s）来抓取这个端点
  3. 这些指标存储在 Prometheus 中，可通过 PromQL 查询——Grafana 从中读取数据绘制仪表盘

Prometheus 定义了四种核心指标类型，在 RAG 监控中都会用到：

指标类型语义RAG 示例
Counter（计数器）只增不减的累计值总请求数、总错误数、总 token 消耗量
Gauge（仪表盘）可增可减的瞬态值当前活跃连接数、知识库文档数、队列大小
Histogram（直方图）观测值的分布统计请求延迟分布、检索结果数分布、chunk 大小分布
Summary（摘要）类似 Histogram，但计算分位数P50/P95/P99 延迟、token 消耗分位数

2.2 创建 metrics.py —— RAG 指标定义

在 rag-service/app/metrics.py 中定义 RAG 服务的所有自定义指标：

"""
app/metrics.py — RAG 服务的 Prometheus 指标定义

使用 prometheus_client 库暴露自定义指标。
这些指标在 /metrics 端点被 Prometheus 定期抓取。
"""

from prometheus_client import Counter, Histogram, Gauge, generate_latest, REGISTRY
from typing import Awaitable, Callable
from fastapi import Request, Response
import time

# ==================== 核心业务指标 ====================

# 请求计数器（按端点和方法区分）
rag_requests_total = Counter(
    "rag_requests_total",
    "RAG 服务总请求数",
    labelnames=["endpoint", "method", "status"],
)

# 请求延迟直方图（毫秒）—— 关注 P50/P95/P99
rag_request_duration_ms = Histogram(
    "rag_request_duration_ms",
    "RAG 请求延迟（毫秒）",
    labelnames=["endpoint"],
    buckets=[50, 100, 200, 500, 1000, 2000, 5000, 10000, 30000, 60000],
)

# 错误计数器（按错误类型区分）
rag_errors_total = Counter(
    "rag_errors_total",
    "RAG 服务错误总数",
    labelnames=["endpoint", "error_type"],
)

# ==================== RAG 专有指标 ====================

# 检索结果数分布（每次检索返回了多少条结果）
rag_retrieval_chunks = Histogram(
    "rag_retrieval_chunks",
    "检索返回的 chunk 数",
    labelnames=["query_type"],
    buckets=[0, 1, 2, 3, 5, 10, 20, 50],
)

# 检索分数分布（检索结果的平均分数——质量退化检测用）
rag_retrieval_score = Gauge(
    "rag_retrieval_score",
    "检索结果的平均分数",
    labelnames=["query_type"],
)

# Token 消耗计数器（按模型区分）
rag_tokens_total = Counter(
    "rag_tokens_total",
    "LLM token 消耗总量",
    labelnames=["model", "token_type"],  # token_type: prompt | completion | total
)

# 拒答计数器（记录 RAG 系统拒答的频率）
rag_refusals_total = Counter(
    "rag_refusals_total",
    "拒答次数",
    labelnames=["reason"],  # reason: no_context | low_confidence | policy
)

# 知识库文档数（Gauge，因为文档数可增可减）
rag_knowledge_docs = Gauge(
    "rag_knowledge_docs",
    "知识库中文档总数",
)

# LLM 调用延迟（单独的直方图，便于分析 LLM 费用曲线）
rag_llm_duration_ms = Histogram(
    "rag_llm_duration_ms",
    "LLM API 调用延迟（毫秒）",
    labelnames=["provider", "model"],
    buckets=[100, 500, 1000, 2000, 5000, 10000, 20000, 30000, 60000],
)

# ==================== FastAPI 中间件 ====================

async def metrics_middleware(request: Request, call_next: Callable) -> Response:
    """
    FastAPI 中间件：自动记录请求指标。

    用法：app.middleware("http")(metrics_middleware)

    对每个请求自动记录：
    - 请求总数（按 endpoint + method + status 区分）
    - 请求延迟直方图
    - 5xx 错误计数
    """
    start_time = time.time()
    response = await call_next(request)
    duration_ms = int((time.time() - start_time) * 1000)

    # 提取端点路径（用于 label）
    endpoint = request.url.path
    method = request.method
    status = str(response.status_code)

    # 记录请求计数
    rag_requests_total.labels(endpoint=endpoint, method=method, status=status).inc()

    # 记录延迟
    rag_request_duration_ms.labels(endpoint=endpoint).observe(duration_ms)

    # 记录 5xx 错误
    if response.status_code >= 500:
        rag_errors_total.labels(
            endpoint=endpoint,
            error_type="server_error",
        ).inc()
    elif response.status_code == 429:
        rag_errors_total.labels(
            endpoint=endpoint,
            error_type="rate_limited",
        ).inc()

    return response


def get_metrics() -> str:
    """
    返回 Prometheus 格式的指标文本。

    通过 /metrics 端点返回给 Prometheus 抓取。
    """
    return generate_latest(REGISTRY).decode("utf-8")

为什么用 Histogram 而不用 Summary？

Prometheus 的 Histogram 和 Summary 都可以计算分位数（P50/P95/P99）。但在 RAG 场景中，推荐使用 Histogram：

  • Histogram 是可聚合的：你可以把多个实例的 Histogram 加起来计算全局 P95——Summary 做不到
  • Histogram 支持灵活的桶边界：RAG 请求延迟范围很广（从 200ms 到 30s），你可以自定义桶来精确捕捉分布
  • Histogram 稍贵（每个桶一个计数器），但在 10 个桶的水平，成本可忽略

2.3 在 RAG Pipeline 中埋点

接下来，在 rag_pipeline.py 中把实际业务指标埋进去——这是监控体系中最有价值的部分：

"""
app/rag_pipeline.py — RAG 管线（含监控指标埋点）

在 L13 代码基础上，加入 Prometheus 指标记录。
改动集中在以下几个方法中：
- retrieve(): 记录检索结果数、检索分数
- ask(): 记录 LLM 延迟、token 消耗、拒答事件
"""

import time
import logging
from typing import Optional
from openai import OpenAI

from .config import settings
from .metrics import (
    rag_retrieval_chunks,
    rag_retrieval_score,
    rag_tokens_total,
    rag_refusals_total,
    rag_llm_duration_ms,
)
from .chat_history import SlidingWindowHistory
from .context_injection import resolve_references
from .model_adapter import create_chat_adapter
from .logger import get_logger

logger = get_logger(__name__)


class RAGPipeline:
    """RAG 管线 —— 含监控指标埋点"""

    def __init__(self):
        self.adapter = create_chat_adapter(
            settings.llm_provider,
            settings.llm_model,
        )
        self._sessions: dict[str, SlidingWindowHistory] = {}
        self.doc_count = 0
        self.is_ready = False
        logger.info("RAGPipeline 初始化完成",
                     provider=settings.llm_provider,
                     model=settings.llm_model)

    def retrieve(self, query: str, top_k: int = 5,
                 score_threshold: float = 0.0) -> list[dict]:
        """
        检索 —— 记录检索结果数和平均分数到 Prometheus。
        """
        start = time.time()
        results = self._search_vector_db(query, top_k, score_threshold)
        latency = int((time.time() - start) * 1000)

        # 📊 记录检索结果数分布
        rag_retrieval_chunks.labels(query_type="general").observe(len(results))

        # 📊 记录检索平均分数
        if results:
            avg_score = sum(r.get("score", 0) for r in results) / len(results)
            rag_retrieval_score.labels(query_type="general").set(avg_score)

        logger.info("检索完成", query=query, results=len(results),
                     latency_ms=latency, avg_score=avg_score if results else 0)
        return results

    def ask(self, query: str, session_id: Optional[str] = None,
            user_id: str = "anonymous",
            temperature: Optional[float] = None,
            max_tokens: Optional[float] = None) -> dict:
        """
        完整 RAG 问答 —— 记录 LLM 延迟、token、拒答等指标。
        """
        start = time.time()

        # 1. 获取/创建会话
        history = self._get_or_create_session(session_id, user_id)

        # 2. 指代消解
        context_text = history.get_recent_context(n=2)
        pronouns = {"它", "该", "这个", "那个", "上述", "该产品"}
        if any(p in query for p in pronouns):
            augmented_query = resolve_references(query, context_text)
        else:
            augmented_query = query

        # 3. 检索
        results = self.retrieve(augmented_query, top_k=settings.retrieval_top_k)

        # 4. 如果无结果，记录拒答并返回
        if not results:
            rag_refusals_total.labels(reason="no_context").inc()
            # ... 拒答逻辑 ...

        # 5. 构建 context 并调用 LLM
        context_str = self._format_context(results)

        # 6. 调用 LLM —— 单独记录 LLM 延迟
        llm_start = time.time()
        try:
            answer = self.adapter.chat(messages, temperature=temperature or 0.7)
        except Exception as e:
            answer = f"抱歉，模型调用出错了：{str(e)}"
            from .metrics import rag_errors_total
            rag_errors_total.labels(endpoint="/ask", error_type="llm_error").inc()
            logger.error("LLM 调用失败", error=str(e))
        llm_duration = int((time.time() - llm_start) * 1000)

        # 📊 记录 LLM 调用延迟
        rag_llm_duration_ms.labels(
            provider=settings.llm_provider,
            model=settings.llm_model,
        ).observe(llm_duration)

        # 📊 记录 token 消耗（粗略估计）
        prompt_tokens = len(query) + len(context_str)
        completion_tokens = len(answer)
        rag_tokens_total.labels(
            model=settings.llm_model,
            token_type="prompt",
        ).inc(prompt_tokens)
        rag_tokens_total.labels(
            model=settings.llm_model,
            token_type="completion",
        ).inc(completion_tokens)
        rag_tokens_total.labels(
            model=settings.llm_model,
            token_type="total",
        ).inc(prompt_tokens + completion_tokens)

        # 7. 更新历史
        history.add_message("user", query)
        history.add_message("assistant", answer)

        latency_ms = int((time.time() - start) * 1000)

        return {
            "answer": answer,
            "session_id": session_id or id(history),
            "model": self.adapter.get_model_info()["model"],
            "latency_ms": latency_ms,
            "llm_duration_ms": llm_duration,
            "prompt_tokens": prompt_tokens,
            "completion_tokens": completion_tokens,
            "refusal": not bool(results),
        }

⚠️ Token 计数：估计值 vs 精确值

上面的例子用 len(text) 估算 token 数量——这不精确。在生产环境中，你应该用 tiktoken 库精确计数：

import tiktoken
encoding = tiktoken.encoding_for_model("gpt-4o-mini")
prompt_tokens = len(encoding.encode(prompt_text))

如果模型 SDK 直接返回 usage.prompt_tokens（如 OpenAI API 的响应中就有），优先使用 API 返回的精确值。本课末尾的动手实践环节会要求你改为精确计数。

  • 单位后缀：_duration_ms（毫秒）、_total（计数器总数）、_bytes（字节）

4.3 关键 PromQL 查询详解

监控 RAG 系统最常用的几个 PromQL 查询——理解它们，你就掌握了 80% 的日常监控需求：

查询目标PromQL 查询解释
每秒请求数（QPS） rate(rag_requests_total[endpoint="/ask"][5m]) 过去 5 分钟 /ask 接口每秒平均请求数。Counter → rate() 是最常用模式。
P95 延迟 histogram_quantile(0.95, rate(rag_request_duration_ms_bucket[5m])) 过去 5 分钟内，95% 的请求延迟低于这个值。比平均延迟更有意义的指标。

Token 消耗趋势 rate(rag_tokens_total{token_type="total", model="gpt-4o-mini"}[1h]) 每小时 token 消耗速率——费用监控的基础。
服务可用性 up{job="rag-api"} Prometheus 内置指标：1 表示目标可达，0 表示不可达。所有 job 的 up 都 =1 才表示系统健康。
容器内存使用 container_memory_usage_bytes{container_label_com_docker_compose_service="rag-api"} cAdvisor 提供的容器级内存指标。label 匹配 Docker Compose 服务名。

PromQL 学习建议

PromQL 是用好 Prometheus 的"门槛"。本课不会深入覆盖所有语法，但建议你花 30 分钟阅读 PromQL 官方基础教程。掌握以下四个模式就能应对 90% 的 RAG 监控场景：

  1. rate(counter[5m]) —— 将 Counter 转换为每秒速率
  2. histogram_quantile(0.95, rate(...)) —— 从 Histogram 计算 P95
  3. sum(rate(...)) by (label) —— 按 label 聚合
  4. gauge > threshold —— 阈值判断（用于告警）

第五部分：LLM 调用链追踪 —— Langfuse

指标和日志可以告诉你"哪里出了问题"，但要理解为什么 LLM 给出了一个糟糕的回答，你需要看到每次 LLM 调用的完整上下文——输入了什么 prompt、返回了什么、用了多少 token、调用了哪个模型。

Langfuse 是一个开源（MIT）的 LLM 可观测性平台，专门解决这个问题。

5.3 集成 Langfuse 到 RAG Pipeline

关键代码 Langfuse Python SDK 提供两种集成方式：

  1. 自动装饰器 @observe()：自动捕获函数的输入输出和耗时——适用于函数级追踪
  2. 手动创建 Traces 和 Spans：更精确的控制——适用于自定义追踪结构

新建 app/tracing.py 文件：

"""
app/tracing.py — Langfuse LLM 调用链追踪

为 RAG 管线的关键环节添加追踪能力：
- 每次 /ask 请求 = 一个 Trace
- 检索阶段 = 一个 Span
- LLM 调用阶段 = 一个 Generation（特殊的 Span，记录 prompt/completion/tokens）
- 质量评估 = 一个 Score

Langfuse SDK 初始化：https://langfuse.com/docs/sdk/python
"""

import os
import logging
from functools import wraps
from typing import Optional, Any
from .config import settings

logger = logging.getLogger(__name__)

# ----- 延迟初始化（避免 Langfuse 未就绪时阻塞应用） -----
_langfuse = None


def get_langfuse():
    """
    获取 Langfuse 客户端实例（延迟初始化）。

    如果 Langfuse 未配置，返回 None——追踪被优雅跳过。
    这样本地开发时不启动 Langfuse 也不影响业务。
    """
    global _langfuse
    if _langfuse is not None:
        return _langfuse

    # 检查环境变量
    public_key = os.getenv("LANGFUSE_PUBLIC_KEY", "")
    secret_key = os.getenv("LANGFUSE_SECRET_KEY", "")
    host = os.getenv("LANGFUSE_HOST", "http://localhost:3000")

    if not public_key or not secret_key:
        logger.info("Langfuse 未配置（缺少 LANGFUSE_PUBLIC_KEY/LANGFUSE_SECRET_KEY），跳过追踪")
        _langfuse = None
        return None

    try:
        from langfuse import Langfuse
        _langfuse = Langfuse(
            public_key=public_key,
            secret_key=secret_key,
            host=host,
        )
        logger.info("Langfuse 客户端初始化成功", extra={"host": host})
        return _langfuse
    except Exception as e:
        logger.warning("Langfuse 初始化失败（不影响服务运行）", extra={"error": str(e)})
        return None


def trace_rag_query(func):
    """
    装饰器：为 RAG 查询方法添加 Langfuse 追踪。

    用法：
        @trace_rag_query
        def ask(self, query, ...):
            ...

    自动创建一个 Trace，记录：
    - 输入：用户查询
    - 输出：RAG 回答
    - 延迟
    - 额外属性（模型、token 数等）
    """

    @wraps(func)
    def wrapper(self, *args, **kwargs):
        langfuse = get_langfuse()
        query = kwargs.get("query", args[0] if args else "unknown")
        session_id = kwargs.get("session_id", "anonymous")

        # 创建 Trace
        trace = None
        if langfuse:
            trace = langfuse.trace(
                name="rag_query",
                input=query,
                session_id=session_id,
                metadata={
                    "user_id": kwargs.get("user_id", "anonymous"),
                    "provider": settings.llm_provider,
                    "model": settings.llm_model,
                },
            )

        # 执行原始方法
        result = func(self, *args, **kwargs)

        # 记录结果到 trace
        if trace and result:
            trace.update(
                output=result.get("answer", ""),
                metadata={
                    **trace.metadata,
                    "latency_ms": result.get("latency_ms", 0),
                    "refusal": result.get("refusal", False),
                    "retrieval_count": len(result.get("retrieval_results", [])),
                },
            )

        return result

    return wrapper


def trace_retrieval(langfuse_trace, query: str, results: list):
    """
    为检索阶段创建一个 Span。

    参数：
        langfuse_trace: Langfuse Trace 实例
        query: 检索使用的查询
        results: 检索结果列表

    返回：检索 Span
    """
    if not langfuse_trace:
        return None

    span = langfuse_trace.span(
        name="retrieval",
        input=query,
        metadata={
            "result_count": len(results),
            "avg_score": sum(r.get("score", 0) for r in results) / len(results) if results else 0,
        },
    )
    span.end(
        output=f"返回 {len(results)} 条结果",
    )
    return span


def trace_llm_call(langfuse_trace, provider: str, model: str,
                   messages: list, response: str,
                   prompt_tokens: int, completion_tokens: int):
    """
    为 LLM 调用创建一个 Generation。

    Generation 是 Langfuse 的特殊 Span 类型，专门用于 LLM 调用。
    Langfuse 会自动统计 token 消耗和费用估算。
    """
    if not langfuse_trace:
        return None

    generation = langfuse_trace.generation(
        name="llm_call",
        model=model,
        model_parameters={
            "provider": provider,
            "temperature": 0.7,
        },
        input=messages,
        output=response,
        usage={
            "prompt_tokens": prompt_tokens,
            "completion_tokens": completion_tokens,
            "total_tokens": prompt_tokens + completion_tokens,
        },
    )
    generation.end()
    return generation


def score_answer(langfuse_trace, name: str, value: float, comment: str = ""):
    """
    为追踪添加评分。

    Langfuse 支持人工评分和模型评分（LLM-as-a-Judge）。
    你可以在这里记录 L7 的 Faithfulness、Answer Relevancy 等指标。
    """
    if not langfuse_trace:
        return

    langfuse_trace.score(
        name=name,
        value=value,
        comment=comment,
    )

这种"树形追踪结构"让你可以一眼看出：检索花了多少时间、LLM 调用花了多少时间、token 用了多少、以及最终回答质量。这是排查"为什么回答差"的核心工具。

第六部分：日志聚合 —— Loki + Promtail

L13 已经实现了 JSON 结构化日志（每行日志是一个 JSON 对象）。但日志分散在多个容器中——当你要排查问题时，需要跨容器搜索日志。

Loki 是 Grafana 生态的日志聚合系统，专为 Prometheus 用户设计。它不索引日志内容，只索引标签（label），因此比 ELK 更轻量、更便宜。

6.1 Loki vs ELK

特性LokiELK（Elasticsearch + Logstash + Kibana）
索引策略只索引 label（标签）全文索引
存储效率高（不索引内容）低（全文索引占大量磁盘）
与 Grafana 集成原生需要额外配置
查询语言LogQL（类似 PromQL）Query DSL
查询速度标签过滤快，内容搜索慢全字段索引，各种查询都快
适合场景Kubernetes/Docker 日志、结构化日志需要全文检索的审计日志

对于 RAG 系统的 JSON 结构化日志，Loki 是更轻量、更经济的选择——因为你的日志已经是 JSON 结构，label 足够定位问题，不需要 ELK 的全文索引能力。

8.1 Alertmanager 配置

创建 alertmanager/alertmanager.yml：

# alertmanager/alertmanager.yml — Alertmanager 告警通知配置
#
# 当 Prometheus 的告警规则被触发时，Alertmanager 负责：
# 1. 分组（Grouping）：相同类型的告警合并为一条通知——避免告警风暴
# 2. 抑制（Inhibition）：高级别告警自动抑制低级别告警
# 3. 路由（Routing）：不同严重级别的告警发到不同的接收者

global:
  # 所有通知的默认参数
  resolve_timeout: 5m

route:
  # 根路由——所有告警先进入这里
  group_by: ["alertname", "severity"]  # 按告警名和严重级别分组
  group_wait: 30s           # 同一组告警等待 30s 再发送——收集更多上下文
  group_interval: 5m        # 同一组告警的发送间隔
  repeat_interval: 4h       # 告警未恢复时，每 4 小时重新通知一次

  # 根据严重级别路由到不同接收者
  routes:
    - receiver: "critical-webhook"
      match:
        severity: "critical"
      repeat_interval: 1h  # critical 级别重复频率更高

    - receiver: "warning-webhook"
      match:
        severity: "warning"

    - receiver: "info-webhook"
      match:
        severity: "info"

receivers:
  - name: "critical-webhook"
    webhook_configs:
      - url: "${WEBHOOK_CRITICAL_URL}"  # 飞书/钉钉/企微 webhook
        send_resolved: true

  - name: "warning-webhook"
    webhook_configs:
      - url: "${WEBHOOK_WARNING_URL}"
        send_resolved: true

  - name: "info-webhook"
    webhook_configs:
      - url: "${WEBHOOK_INFO_URL}"
        send_resolved: false  # info 级别的告警恢复不通知

# 抑制规则：如果已经有 critical 告警正在触发，
# 则自动抑制同一服务上的 warning 和 info 告警
inhibit_rules:
  - source_match:
      severity: "critical"
    target_match:
      severity: "warning"
    equal: ["alertname"]

  - source_match:
      severity: "critical"
    target_match:
      severity: "info"
    equal: ["alertname"]

第九部分：检索质量退化自动检测

这是最容易被忽视但影响最大的监控维度。大部分团队只看 CPU/内存/错误率——但检索质量下降是"慢性病"：

  • 知识库更新了，但 chunk 质量变差了
  • Embedding 模型切换了，但新模型不适合你的数据
  • Chroma 索引因为数据量增长而性能急剧下降
  • 某个文档被误删了，导致特定问题再也检索不到

这些问题不会立即导致 500 错误，但会让用户觉得"RAG 系统变笨了"——这种负面影响是致命的。

9.1 基于基线对比的质量巡检脚本

创建 scripts/quality_check.py，基于 L7 的评估框架做定期检查：

"""
scripts/quality_check.py — RAG 检索质量巡检脚本

定期运行这个脚本（通过 cron/GitHub Actions），检查：
1. 检索结果的 score 分布是否正常
2. Golden Test Set 的召回率是否下降
3. 针对预定义问题的 Faithfulness 评分是否变化

当质量指标低于基线时，发出告警。
这比"用户投诉了才知道"要主动得多。
"""

import json
import sys
import os
from datetime import datetime

# 确保可以从父目录导入
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))

from app.rag_pipeline import rag_pipeline
from app.config import settings


# ==================== 配置 ====================

# Golden Test Set —— 一组你知道正确答案的查询
GOLDEN_QUERIES = [
    {"query": "什么是 RAG？", "expected_keywords": ["检索", "增强", "生成"]},
    {"query": "Chroma 是什么数据库？", "expected_keywords": ["向量", "数据库"]},
    {"query": "向量检索有哪些方式？", "expected_keywords": ["相似度", "ANN", "近似"]},
]

# 质量基线（从最近一次正常运行的巡检结果更新）
QUALITY_BASELINE = {
    "avg_score": 0.55,          # 平均检索分数
    "min_recall@5": 0.7,       # 最低召回率（Top-5）
    "min_faithfulness": 0.8,   # 最低 Faithfulness 评分
}


# ==================== 质量检查 ====================

def check_retrieval_scores() -> dict:
    """
    检查检索分数的分布是否正常。

    正常范围通常 0.3-0.9（取决于 embedding 模型和数据集）。
    如果平均分突然大幅下降（>20%），说明可能有数据异常。
    """
    scores = []
    for item in GOLDEN_QUERIES:
        results = rag_pipeline.retrieve(item["query"], top_k=5)
        if results:
            scores.append(sum(r["score"] for r in results) / len(results))

    if not scores:
        return {"status": "error", "message": "检索无返回结果", "avg_score": 0}

    avg_score = sum(scores) / len(scores)
    baseline = QUALITY_BASELINE["avg_score"]

    degradation = (baseline - avg_score) / baseline

    return {
        "status": "ok" if degradation < 0.2 else "degraded",
        "avg_score": round(avg_score, 4),
        "baseline_avg_score": baseline,
        "degradation_pct": round(degradation * 100, 2),
        "message": (
            f"检索评分正常（avg={avg_score:.3f}）"
            if degradation < 0.2
            else f"⚠️ 检索评分下降 {degradation*100:.1f}%（baseline={baseline}，current={avg_score}）"
        ),
    }


def check_recall_at_k() -> dict:
    """
    检查 Golden Test Set 的召回率（Recall@K）。

    对每个已知答案的问题，检查 Top-5 检索结果中
    是否包含期望的关键词。
    """
    total = len(GOLDEN_QUERIES)
    found = 0

    for item in GOLDEN_QUERIES:
        results = rag_pipeline.retrieve(item["query"], top_k=5)
        if not results:
            continue

        # 检查结果中是否包含期望关键词
        all_text = " ".join(r["text"] for r in results)
        if any(kw in all_text for kw in item["expected_keywords"]):
            found += 1

    recall = found / total if total > 0 else 0
    baseline = QUALITY_BASELINE["min_recall@5"]

    return {
        "status": "ok" if recall >= baseline else "degraded",
        "recall@5": round(recall, 4),
        "baseline_recall@5": baseline,
        "message": (
            f"Recall@5={recall:.1%}，高于基线 {baseline:.0%}"
            if recall >= baseline
            else f"⚠️ Recall@5={recall:.1%}，低于基线 {baseline:.0%}"
        ),
    }


def check_embedding_consistency() -> dict:
    """
    检查 Embedding 的一致性（向量空间是否发生漂移）。

    方法：对同一段文本计算两次 embedding，看向量是否一致。
    如果 embedding 模型被切换或更新，两天的 embedding 可能
    不在同一个向量空间中——导致检索结果质量下降。
    """
    test_text = "这是一个用于检测 embedding 一致性的测试句子。"

    embedding1 = rag_pipeline._get_embedding(test_text)
    embedding2 = rag_pipeline._get_embedding(test_text)

    if not embedding1 or not embedding2:
        return {"status": "error", "message": "无法获取 embedding"}

    # 计算余弦相似度（两次理论上应该完全相同）
    dot = sum(a * b for a, b in zip(embedding1, embedding2))
    norm1 = sum(a * a for a in embedding1) ** 0.5
    norm2 = sum(b * b for b in embedding2) ** 0.5
    similarity = dot / (norm1 * norm2) if norm1 * norm2 > 0 else 0

    return {
        "status": "ok" if similarity > 0.99 else "warning",
        "embedding_similarity": round(similarity, 6),
        "message": (
            f"Embedding 一致性正常（cosine={similarity:.6f}）"
            if similarity > 0.99
            else f"⚠️ Embedding 不一致！cosine={similarity:.6f}，可能模型已变更"
        ),
    }


# ==================== 主流程 ====================

def run_quality_check() -> dict:
    """执行全部质量检查，输出结果并决定是否告警。"""
    results = {
        "timestamp": datetime.now().isoformat(),
        "provider": settings.embedding_provider,
        "model": settings.embedding_model,
        "doc_count": rag_pipeline.doc_count,
    }

    # 检索分数检查
    results["retrieval_scores"] = check_retrieval_scores()

    # 召回率检查
    results["recall"] = check_recall_at_k()

    # Embedding 一致性检查
    results["embedding"] = check_embedding_consistency()

    # 综合状态
    checks = [
        results["retrieval_scores"]["status"],
        results["recall"]["status"],
        results["embedding"]["status"],
    ]
    results["overall_status"] = "healthy" if all(s == "ok" for s in checks) else "degraded"
    results["alert"] = any(s in ("degraded", "error", "warning") for s in checks)

    return results


if __name__ == "__main__":
    report = run_quality_check()
    print(json.dumps(report, ensure_ascii=False, indent=2))

    if report.get("alert"):
        print("\n🚨 质量告警！需要人工介入检查。", file=sys.stderr)

质量检测的"信号"与"噪声"

质量检测最怕什么？假告警——告警太多，团队就不看了。以下是减少假告警的策略：

  • 不要用绝对值："平均分 < 0.3" 这个阈值因数据集而异。更好的是相对变化率："比 7 天均线下降 20%"
  • 黄金数据集要精心设计：你的 Golden Test Set 必须是"有确定正确答案且覆盖主要业务场景"的——不要用模糊的问题做质检
  • 告警要有可操作性：每条告警要说明"发生了什么"、"影响范围"、"建议操作"——让值班的人不迷茫
  • 定期更新基线：知识库内容增长后，检索分数自然会变化。基线需要自适应更新

故障诊断：常见问题与排查

❌ 问题一：调用 /metrics 返回 404

原因：metrics 路由未注册，或中间件未应用。

排查步骤：

  1. 检查 app/main.py 中是否导入了 metrics_middleware 和 get_metrics
  2. 确认有 app.middleware("http")(metrics_middleware) 这行代码
  3. 确认有 @app.get("/metrics") 路由装饰器
  4. 检查是否有其他路由覆盖了 /metrics 路径（比如 Nginx 配置）

❌ 问题二：Grafana 中 Prometheus 数据源显示 "Unreachable"

原因：Grafana 容器无法通过 http://prometheus:9090 访问 Prometheus。

排查步骤：

  1. 检查两个服务是否在同一个 Docker 网络（rag-network）中
  2. 进入 Grafana 容器测试连接：docker compose exec grafana sh -c "wget -qO- http://prometheus:9090/api/v1/query?query=up"
  3. 检查 grafana/datasources/prometheus-datasource.yml 中的 URL 是否正确
  4. 如果用了 localhost 而不是 Docker 服务名——在 Docker 网络内部，服务名才是正确的 host

❌ 问题三：Promtail 无法采集容器日志，Loki 中没有数据

原因：Promtail 无法访问 Docker Socket 或配置有误。

排查步骤：

  1. 检查 Promtail 日志：docker compose logs promtail——是否有 error reading docker socket 之类的错误
  2. 确认 docker.sock 已正确挂载：docker compose exec promtail ls -la /var/run/docker.sock——应该显示 socket 文件
  3. 检查 promtail 配置中的 docker_sd_configs 是否拼写正确
  4. 在 Grafana Explore 中手动查询 {job="docker"} 而不是 {service="rag-api"}——确认 label 是否正确

❌ 问题四：Langfuse 启动后页面白屏或报 DB 错误

原因：Langfuse 的 PostgreSQL 数据库未就绪或 schema 迁移失败。

排查步骤：

  1. 检查 PostgreSQL 是否健康：docker compose logs langfuse-db——应有 "database system is ready to accept connections"
  2. 检查 Langfuse 日志：docker compose logs langfuse-server——寻找 migration 错误
  3. 确认 DATABASE_URL 环境变量中的用户名/密码与 PostgreSQL 配置一致
  4. 如果持续失败，尝试重建：docker compose down -v langfuse-db && docker compose up -d langfuse-db（⚠️ 会丢失已存储的追踪数据）

❌ 问题五：告警触发但没有收到通知

原因：Alertmanager 配置或 webhook URL 问题。

排查步骤：

  1. 检查 Alertmanager 日志：docker compose logs alertmanager——是否有 "Error sending webhook" 错误
  2. 在 Alertmanager UI（http://localhost:9093）查看告警是否确实到达了 Alertmanager
  3. 手动测试 webhook URL：curl -X POST $WEBHOOK_CRITICAL_URL -H 'Content-Type: application/json' -d '{"msg":"test"}'
  4. 检查 .env 中的 webhook URL 是否被正确注入到 Alertmanager 容器（注意 ${WEBHOOK_CRITICAL_URL} 语法）
