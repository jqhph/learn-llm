第一部分：云服务器选型与创建

第一步是选择并创建一台云服务器。国内主流的云服务商是阿里云和腾讯云，操作流程大同小异。

1.1 实例规格选择

根据 RAG 服务的资源需求，给出以下推荐配置：

为什么推荐 8GB 内存？

RAG 服务栈包含：RAG API（Python 进程 ~200MB）、Chroma（~1-2GB，取决于数据量）、Redis（~100MB）、Nginx（~50MB），加上系统开销。4GB 勉强可跑但容易 OOM，8GB 是安全线。如果你使用本地 Embedding 模型（如 BGE-M3 通过 Ollama），还需要额外 2-4GB。

1.2 阿里云 ECS 创建流程

以下步骤登录阿里云控制台完成（腾讯云 CVM 类似，见下节对比）：

  2. 配置项：

    • 镜像：Ubuntu 22.04 LTS（64 位）—— 本课所有命令基于 Ubuntu

    • 专有网络 VPC：使用默认 VPC 即可

1.4 SSH 密钥登录配置

密码登录既不安全又不方便。SSH 密钥是标准做法：

# 1. 在本地生成密钥对（如果还没有）
ssh-keygen -t ed25519 -C "your-email@example.com"
# 执行后默认在 ~/.ssh/id_ed25519（私钥）和 ~/.ssh/id_ed25519.pub（公钥）

# 2. 查看公钥并复制
cat ~/.ssh/id_ed25519.pub
# 输出：ssh-ed25519 AAAAC3... your-email@example.com

# 3. 创建云服务器时，选择"密钥对"方式，粘贴公钥
#    或者在已有服务器上手动添加：
ssh root@<服务器IP> "mkdir -p ~/.ssh && echo 'ssh-ed25519 AAAAC3... your-email@example.com' >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"

SSH 连接测试

# 创建完成后，通过 SSH 连接到云服务器
ssh root@<服务器公网IP>

# 第一次连接会提示确认主机指纹，输入 yes
# 成功后你将看到 Ubuntu 的欢迎信息：
# Welcome to Ubuntu 22.04.4 LTS (GNU/Linux 5.15.0-... x86_64)

第二部分：服务器环境初始化

新服务器是"毛坯房"——需要安装 Docker、配置镜像加速、设置防火墙。

2.1 Docker 安装（阿里云镜像加速）

国内下载 Docker 可能很慢，使用阿里云镜像源加速：

# ---- 安装 Docker（阿里云镜像） ----
curl -fsSL https://get.docker.com | bash -s docker --mirror Aliyun

# 验证安装
docker --version
# 预期：Docker version 26.x.x

# ---- 配置 Docker 镜像加速 ----
# 1. 登录阿里云 → 容器镜像服务 ACR → 镜像中心 → 镜像加速器
#    你会得到一个专属于你的加速地址：https://xxxx.mirror.aliyuncs.com

# 2. 配置 daemon.json
sudo mkdir -p /etc/docker
sudo tee /etc/docker/daemon.json <<'EOF'
{
  "registry-mirrors": ["https://你的专属加速地址.mirror.aliyuncs.com"],
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "exec-opts": ["native.cgroupdriver=systemd"]
}
EOF

# 3. 重启 Docker 使配置生效
sudo systemctl daemon-reload
sudo systemctl restart docker

# 4. 验证镜像加速生效
docker info | grep "Registry Mirrors"
# 预期：Registry Mirrors: https://你的专属加速地址.mirror.aliyuncs.com

Docker 日志限制为什么重要？

默认情况下，Docker 容器的日志文件不会自动轮转，会一直增长。一个 RAG API 容器一天可能产生数百 MB 日志——如果不限制，一个月后你的 40 GB 系统盘就被日志塞满了，服务直接挂掉。max-size: 10m 和 max-file: 3 的意思是：每个容器最多保留 3 个日志文件，每个不超过 10MB，总计不超过 30MB。这是生产必备配置。

2.2 Docker Compose 安装

Docker Compose v2 已集成到 Docker CLI 中（docker compose 命令）。如果上面安装了新版 Docker，它已经可用：

# 验证 Docker Compose

2.3 系统优化

为了让服务器稳定运行 RAG 服务，做几项关键优化：

# ---- 1. 降低 swappiness（减少磁盘交换，提升性能） ----
# Docker 容器大量使用内存，如果系统频繁 swap 到磁盘，性能急剧下降
echo "vm.swappiness=10" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p

# ---- 2. 增加文件描述符限制（Docker 需要大量文件句柄） ----
echo -e "root soft nofile 65536\nroot hard nofile 65536" | sudo tee -a /etc/security/limits.conf
# 重新登录后生效（或重启）

# ---- 3. 配置 UFW 防火墙 ----
sudo ufw allow 22/tcp      # SSH
sudo ufw allow 80/tcp      # HTTP（Let's Encrypt 验证）
sudo ufw allow 443/tcp     # HTTPS
sudo ufw --force enable    # 启用防火墙
sudo ufw status            # 验证规则
# 预期输出类似：
# Status: active
# 22/tcp    ALLOW
# 80/tcp    ALLOW
# 443/tcp   ALLOW

# ---- 4. 配置时区 ----
sudo timedatectl set-timezone Asia/Shanghai
timedatectl | grep "Time zone"
# 预期：Time zone: Asia/Shanghai (CST, +0800)

# ---- 5. 修改主机名（便于识别） ----
sudo hostnamectl set-hostname rag-server
# 重新登录后提示符会变成 root@rag-server:~$

⚠️ 关于 UFW 防火墙 vs 安全组

你的云服务器实际上有两层防火墙：

  • 云平台安全组（第一层）：在云控制台配置，控制从公网进来的流量
  • 服务器 UFW（第二层）：在服务器内部配置，提供深度防御

安全组放行 80/443 后，UFW 的规则作为"兜底"：即使误开放了其他安全组规则，UFW 也会拦截。两层防护都用上，这是纵深防御原则。

第三部分：从本地到云端部署 RAG 服务

环境就绪后，把 L13 的服务栈部署到云上。核心问题：本地的 Docker 镜像和代码怎么到服务器上？有三种方案。

3.1 部署方案对比

方案怎么做优点缺点推荐
方案 A：镜像仓库本地构建镜像 → push 到仓库 → 服务器 pull标准化；CI/CD 友好需要配置镜像仓库⭐⭐⭐ 生产推荐
方案 B：服务器直接构建代码传到服务器 → 服务器上 docker compose build无需仓库；无跨平台问题构建慢；依赖服务器算力⭐⭐ 适合初学
方案 C：rsync 传输产物本地构建 → docker save → scp 镜像 → docker load无需仓库；离线友好传输大文件慢；不好自动化⭐ 应急方案

本课按方案 B + 方案 A的顺序学习：先用方案 B 快速跑通，再升级到方案 A（镜像仓库）为 CI/CD 做准备。

3.2 方案 B：服务器直接构建（快速起步）

# ---- 在本地：将代码推送到 Git 仓库 ----
# 将 L13 的项目代码推送到 GitHub（或 Gitee）
# 我们假设仓库地址为：https://github.com/your-org/rag-service.git

# ---- 在服务器上：克隆代码并构建 ----
ssh root@<服务器IP>

# 1. 创建项目目录
mkdir -p /opt/rag-service
cd /opt/rag-service

# 2. 克隆代码
git clone https://github.com/your-org/rag-service.git .

# 3. 创建 .env 文件（⚠️ 生产环境使用真实 API Key）
cat > .env <<'EOF'
OPENAI_API_KEY=sk-your-real-key
DASHSCOPE_API_KEY=sk-your-dashscope-key
DEEPSEEK_API_KEY=sk-your-deepseek-key
LLM_PROVIDER=openai
LLM_MODEL=gpt-4o-mini
EMBEDDING_PROVIDER=openai
EMBEDDING_MODEL=text-embedding-3-small
REDIS_PASSWORD=strong-password-here
LOG_LEVEL=INFO
LOG_FORMAT=json
EOF

# 4. 构建并启动（服务器是 x86 架构，没问题）
docker compose up -d --build

# 5. 检查状态
docker compose ps
# 预期：所有服务 Up（healthy）

# 6. 直接测试（通过内部端口）
curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"query":"你的知识库里有什么？"}'
# 预期：返回正常回答

❌ 跨平台构建陷阱（ARM → x86）

如果你在 Mac M 系列（ARM 芯片）上开发：

# 本地构建的镜像在服务器上运行会报错：
# "exec format error" 或 "standard_init_linux.go:..."
# 原因：ARM 指令集 ≠ x86 指令集

# 解决方案一：在 Dockerfile 中指定平台
FROM --platform=linux/amd64 python:3.12.4-slim AS builder
FROM --platform=linux/amd64 python:3.12.4-slim

# 解决方案二：用 buildx 构建跨平台镜像
docker buildx build --platform linux/amd64 -t rag-service:latest .

# 解决方案三（最简单）：不要在本地构建，直接把代码传到服务器上构建
# 服务器是 x86 芯片，在服务器上 docker compose build 自然就是 x86 镜像

对于方案 B（服务器直接构建）不存在这个问题——因为代码是纯文本，服务器构建时原生就是 x86。

3.3 方案 A：镜像仓库 + 跨平台构建（CI/CD 准备）

方案 B 虽简单，但有缺点：每次部署都要在服务器上 git pull 再 docker compose build——构建耗时且占服务器资源。真正的生产流程是：本地/CI 构建好镜像 → 推送到镜像仓库 → 服务器拉取启动。

首先，选择一个镜像仓库：

镜像仓库国内速度免费额度适用场景
Docker Hub慢（需配置镜像加速）1 个私有仓库（免费）海外用户、开源项目
阿里云 ACR 个人版极快（与 ECS 同区域）300 个镜像，不限私有国内用户首选
腾讯云 TCR 个人版极快（与 CVM 同区域）无限私有（2026 年政策）腾讯云用户首选
GitHub Container Registry中等（依赖网络）无限公开，有限私有GitHub 深度用户

阿里云 ACR 配置步骤

# ---- 1. 在本地登录阿里云 ACR ----
# 获取 ACR 实例地址（登录阿里云 → 容器镜像服务 ACR → 个人实例）
# 实例地址类似：registry.cn-hangzhou.aliyuncs.com

# 创建固定密码（ACR → 访问凭证 → 设置固定密码）
docker login --username=你的阿里云账号 registry.cn-hangzhou.aliyuncs.com
# 输入密码（固定密码）

# ---- 2. 给镜像打标签并推送 ----
# 在本地（代码目录下）
docker build -t rag-service:latest .

# 标记：格式 <仓库地址>/<命名空间>/<镜像名>:<标签>
docker tag rag-service:latest registry.cn-hangzhou.aliyuncs.com/你的命名空间/rag-service:latest

# 推送
docker push registry.cn-hangzhou.aliyuncs.com/你的命名空间/rag-service:latest

# ---- 3. 在服务器上拉取并运行 ----
ssh root@<服务器IP>

# 登录 ACR
docker login --username=你的阿里云账号 registry.cn-hangzhou.aliyuncs.com
# 输入固定密码

# 拉取镜像
docker pull registry.cn-hangzhou.aliyuncs.com/你的命名空间/rag-service:latest

# 更新 docker-compose.yml：将 build: . 替换为 image: 指向仓库地址
# 然后启动
docker compose up -d

ACR 专用网络加速

如果 ECS 和 ACR 在同一个地域（比如都在"华东 1（杭州）"），Docker 推拉走的是阿里云内网，速度极快（几百 MB 的镜像几秒传输）。跨地域则走公网，可能慢。所以创建 ECS 时选一个同时支持 ACR 的地域。

3.4 更新 docker-compose.yml 支持镜像仓库

为了让 docker-compose.yml 同时支持本地构建和镜像仓库两种模式，做如下调整：

# docker-compose.yml（适配云部署版本）
services:
  rag-api:
    # 支持两种构建方式：
    # 1. 本地构建：build: .
    # 2. 镜像仓库：image: registry.xxx.com/ns/rag-service:latest
    build:
      context: .
      dockerfile: Dockerfile
      args:
        # 在 CI 中由 GitHub Actions 传入
        BUILDPLATFORM: ${BUILDPLATFORM:-linux/amd64}
    image: ${IMAGE_TAG:-rag-service:latest}
    container_name: rag-api
    restart: unless-stopped
    ports:
      - "8000:8000"
    environment:
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      - DASHSCOPE_API_KEY=${DASHSCOPE_API_KEY}
      - DEEPSEEK_API_KEY=${DEEPSEEK_API_KEY}

第四部分：Nginx 反向代理 + HTTPS 证书

现在你的服务跑在 http://服务器IP:8000——既没有域名，也没有 HTTPS。这在生产场景中是不可接受的：HTTP 明文传输会泄露 API Key、对话内容等敏感信息。接下来配置 HTTPS。

4.1 域名准备

  • 使用海外服务器（如新加坡、香港的 ECS）——不需要备案，但延迟稍高
  • 使用 ngrok 或 Cloudflare Tunnel 临时暴露本地服务——适合短期演示

4.2 安装 Certbot + 申请 Let's Encrypt 证书

Let's Encrypt 提供免费 SSL 证书，有效期 90 天，可以自动续期。Certbot 是官方客户端：

# ---- 在服务器上执行 ----
ssh root@<服务器IP>

# 1. 安装 Certbot 和 Nginx 插件
sudo apt update
sudo apt install -y certbot python3-certbot-nginx

# 2. 先确保 Nginx 容器正在运行（为了验证域名）
# 此时你的 Nginx 应该正在监听 80 端口
# 如果 Nginx 容器运行在你的 Docker 网络中，注意端口冲突问题
# 建议先停止 Docker 的 Nginx，用系统安装的 Nginx 走完证书流程：

# 2a. 停掉 Docker 中的 Nginx（稍后重新配置 HTTPS 后再启动）
docker compose stop nginx

# 2b. 安装系统 Nginx（仅用于 Let's Encrypt 验证）
sudo apt install -y nginx

# 2c. 创建一个临时的 Nginx 配置，让域名能访问
sudo tee /etc/nginx/sites-enabled/default <<'EOF'
server {
    listen 80;
    server_name your-domain.com;
    root /var/www/html;
}
EOF
sudo systemctl reload nginx

# 3. 申请证书（certbot --nginx 会自动修改 Nginx 配置）
sudo certbot --nginx -d your-domain.com -d www.your-domain.com

# 交互过程会：
# - 要求输入邮箱（用于续期提醒）
# - 同意 Let's Encrypt 服务条款
# - 选择是否将 HTTP 重定向到 HTTPS（选 "2" Redirect）
# 成功后输出：
# Congratulations! Your certificate and chain have been saved at:
#   /etc/letsencrypt/live/your-domain.com/fullchain.pem
#   /etc/letsencrypt/live/your-domain.com/privkey.pem

HTTP-01 验证原理

Let's Encrypt 通过 HTTP-01 挑战验证你对域名的控制权：

  1. Certbot 要求 Let's Encrypt 签发证书
  2. Let's Encrypt 生成一个随机 token：http://your-domain.com/.well-known/acme-challenge/<token>
  3. Certbot 在 Nginx 中放置这个 token 文件的响应
  4. Let's Encrypt 全球多个节点访问这个 URL——只要能正确返回 token，就证明你控制了该域名
  5. 验证通过后签发证书

所以 80 端口必须对公网开放，且不能有防火墙阻断 Let's Encrypt 的验证服务器访问。

4.3 Nginx HTTPS 配置

证书就绪后，重新配置 Nginx 作为 RAG API 的反向代理（HTTPS）：

# 停止系统 Nginx（证书已获取，不再需要它）
sudo systemctl stop nginx
sudo apt remove -y nginx

# 修改 Docker 中的 Nginx 配置
# nginx.conf — 生产级 HTTPS + 反向代理配置
# nginx.conf — RAG 服务 HTTPS 反向代理（生产版）
upstream rag_api {
    server rag-api:8000;
    keepalive 32;  # 保持长连接，减少握手开销
}

# HTTP → HTTPS 强制重定向
server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$server_name$request_uri;
}

# HTTPS 服务
server {
    listen 443 ssl http2;
    server_name your-domain.com;

    # SSL 证书路径
    ssl_certificate /etc/letsencrypt/live/your-domain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/your-domain.com/privkey.pem;

    # 现代 TLS 配置
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;

    # HSTS（告诉浏览器以后只走 HTTPS）
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

    # 安全头部
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options DENY;
    add_header X-XSS-Protection "1; mode=block";

    # 请求体大小限制（根据你的知识库文档大小调整）
    client_max_body_size 50m;

    # JSON 格式的访问日志
    log_format json escape=json
        '{'
          '"time":"$time_iso8601",'
          '"remote_addr":"$remote_addr",'
          '"method":"$request_method",'
          '"uri":"$uri",'
          '"status":$status,'
          '"body_bytes":$body_bytes_sent,'
          '"request_time":$request_time,'
          '"upstream_addr":"$upstream_addr",'
          '"upstream_response_time":"$upstream_response_time"'
        '}';
    access_log /var/log/nginx/access.log json;
    error_log /var/log/nginx/error.log;

    # ---- RAG API 反向代理 ----
    location /api/ {
        rewrite ^/api(/.*)$ $1 break;
        proxy_pass http://rag_api;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # LLM 生成可能耗时较长（30s+），必须加大超时
        proxy_connect_timeout 10s;
        proxy_read_timeout 120s;
        proxy_send_timeout 30s;

        # WebSocket 支持（如果需要流式输出）
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # 健康检查
    location /health {
        proxy_pass http://rag_api/health;
        proxy_set_header Host $host;
        access_log off;  # 健康检查不记日志，节省磁盘
    }

    # 生产环境建议关闭 API 文档暴露
    location /docs {
        deny all;
        return 404;
    }
    location /redoc {
        deny all;
        return 404;
    }
}

关键变化：

  • HTTP 重定向：所有 80 端口的请求 301 重定向到 HTTPS
  • HSTS：告诉浏览器以后永远用 HTTPS 访问，杜绝一次 HTTP 请求被劫持
  • 安全头部：X-Content-Type-Options / X-Frame-Options / X-XSS-Protection 是防御 XSS 和点击劫持的标准方案
  • SSL 配置：只允许 TLSv1.2 和 TLSv1.3（TLSv1.0/1.1 有安全漏洞，已废弃）
  • 关闭 API 文档：/docs 在生产环境应该关闭——暴露 API schema 等于告诉攻击者你的全部接口

4.5 自动续期（Systemd Timer）

Let's Encrypt 证书有效期为 90 天。手动续期不可接受——必须自动化。Systemd Timer 是比 cron 更现代的方案：

# ---- 创建 systemd service ----
# /etc/systemd/system/certbot-renew.service
sudo tee /etc/systemd/system/certbot-renew.service <<'EOF'
[Unit]
Description=Certbot Automatic Renewal
After=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/bin/certbot renew --post-hook "docker exec rag-nginx nginx -s reload"
EOF

# ---- 创建 systemd timer ----
# /etc/systemd/system/certbot-renew.timer
sudo tee /etc/systemd/system/certbot-renew.timer <<'EOF'
[Unit]
Description=Run certbot renewal twice daily

[Timer]
# 每天凌晨和中午各检查一次（Let's Encrypt 建议）
OnCalendar=*-*-* 00,12:00:00
# 随机延迟 0-1 小时，避免所有服务器同时请求 Let's Encrypt
RandomizedDelaySec=1h
# 如果错过上次触发时间，立即执行一次
Persistent=true

[Install]
WantedBy=timers.target
EOF

# ---- 启用并启动 timer ----
sudo systemctl daemon-reload
sudo systemctl enable --now certbot-renew.timer

# ---- 验证 timer 状态 ----
systemctl status certbot-renew.timer
# 预期：● certbot-renew.timer - Run certbot renewal twice daily
#        Loaded: loaded
#        Active: active (waiting)

# ---- 手动测试续期（不会真的续期，只是演练）----
sudo certbot renew --dry-run
# 预期：Congratulations, all renewals succeeded. ...

为什么用 systemd timer 而不是 cron？

  • 更强的依赖管理：可以声明 After=network-online.target——确保网络就绪后才执行
  • 错过执行的处理：Persistent=true 确保即使服务器在触发时间关机，开机后也会补执行
  • 日志统一：通过 journalctl -u certbot-renew.service 查看续期日志
  • 随机延迟：RandomizedDelaySec=1h 避免服务器集群同时访问 Let's Encrypt 造成 DDoS 嫌疑

第五部分：GitHub Actions CI/CD 流水线

到目前，每次更新代码你都要手动 SSH 上服务器执行 git pull && docker compose up -d --build。这在一个人的项目中还能忍，但一旦团队协作或频繁更新，这就是不归路。需要自动化。

CI/CD = 持续集成 + 持续部署。本课实现的目标：

git push → 自动触发流水线
├─ CI 阶段（在 GitHub Actions Runner 上运行）
│ ├─ 1️⃣ 安装依赖
│ ├─ 2️⃣ 运行 pytest 自动化测试
│ ├─ 3️⃣ 构建 Docker 镜像
│ └─ 4️⃣ 推送镜像到阿里云 ACR
├─ CD 阶段（通过 SSH 在服务器上运行）
│ ├─ 5️⃣ SSH 连接到云服务器
│ ├─ 6️⃣ 从 ACR 拉取最新镜像
│ └─ 7️⃣ docker compose up -d 重启服务
└─ ✅ 部署完成，时间 ≈ 3-5 分钟

5.1 创建 GitHub Actions Workflow

在你的项目根目录创建 .github/workflows/deploy.yml：

# .github/workflows/deploy.yml
# RAG 服务 CI/CD 流水线 —— 自动构建→测试→部署到云服务器

name: Deploy RAG Service

# 触发条件：推送到 main 分支时触发
on:
  push:
    branches: [main]
  # 也支持手动触发（在 GitHub Actions 页面点按钮）
  workflow_dispatch:

# 环境变量（对所有 job 全局可用）
env:
  # 镜像仓库地址
  REGISTRY: registry.cn-hangzhou.aliyuncs.com
  NAMESPACE: your-namespace
  IMAGE_NAME: rag-service

jobs:
  # ========== Job 1：测试和构建 ==========
  build-and-push:
    name: "Build & Push Docker Image"
    runs-on: ubuntu-latest
    # 此 job 中的步骤：
    steps:
      # 1. 检出代码
      - name: Checkout repository
        uses: actions/checkout@v4

      # 2. 设置 Python 环境（用于运行测试）
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'
          cache: 'pip'

      # 3. 安装依赖
      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest  # 测试框架

      # 4. 运行测试（CI 的核心——确保代码没问题才继续）
      - name: Run tests
        run: |
          pytest tests/ -v --timeout=30
        env:
          # 测试环境需要的最小配置
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

      # 5. 登录到阿里云 ACR
      - name: Login to Aliyun ACR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ secrets.ACR_USERNAME }}
          password: ${{ secrets.ACR_PASSWORD }}

      # 6. 构建并推送 Docker 镜像
      - name: Build and push Docker image
        uses: docker/build-push-action@v6
        with:
          context: .
          file: ./Dockerfile
          # 指定平台（确保在 x86 服务器上能跑）
          platforms: linux/amd64
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.NAMESPACE }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.NAMESPACE }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          # 利用缓存加速构建
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ========== Job 2：部署到服务器 ==========
  deploy:
    name: "Deploy to Cloud Server"
    runs-on: ubuntu-latest
    # 等待 build-and-push 完成后才执行
    needs: build-and-push
    steps:
      # 7. SSH 到服务器，执行部署命令
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USERNAME }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          # 连接到服务器后执行的命令
          script: |
            set -e  # 任何命令失败就停止（防止"部分部署"）
            echo "🚀 开始部署 ..."

            # 进入项目目录
            cd /opt/rag-service

            # 拉取最新的 docker-compose.yml（如果改了配置）
            git pull origin main

            # 登录 ACR
            docker login ${{ env.REGISTRY }} \
              -u ${{ secrets.ACR_USERNAME }} \
              -p ${{ secrets.ACR_PASSWORD }}

            # 拉取最新镜像
            docker pull ${{ env.REGISTRY }}/${{ env.NAMESPACE }}/${{ env.IMAGE_NAME }}:latest

            # 更新 IMAGE_TAG 环境变量并重启
            export IMAGE_TAG=${{ env.REGISTRY }}/${{ env.NAMESPACE }}/${{ env.IMAGE_NAME }}:latest
            docker compose down --remove-orphans || true
            docker compose up -d

            # 等待服务就绪
            echo "⏳ 等待服务健康检查..."
            sleep 15

            # 验证部署
            curl -f http://localhost:8000/health || {
              echo "❌ 健康检查失败，回滚！"
              docker compose down
              # 回滚到上一个版本
              docker pull ${{ env.REGISTRY }}/${{ env.NAMESPACE }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
              docker compose up -d
              exit 1
            }

            # 清理旧镜像（保留最近 3 个）
            docker image prune -a --force --filter "until=24h" || true

            echo "✅ 部署成功！"

      # 8. 部署完成后通知（选配）
      - name: Send notification
        if: always()
        uses: actions/github-script@v7
        with:
          script: |
            const status = '${{ job.status }}';
            const message = status === 'success'
              ? '✅ RAG 服务部署成功：https://your-domain.com/health'
              : '❌ RAG 服务部署失败，请检查：https://github.com/${{ github.repository }}/actions';
            console.log(message);
            // 可以在这里接入飞书/钉钉/企业微信 WebHook 通知

# 在 GitHub 仓库 → Settings → Secrets and variables → Actions → New repository secret
# 添加以下 Secrets：

# ---- 服务器 SSH ----
# SERVER_HOST        → 你的服务器公网 IP（如 123.123.123.123）

# OPENAI_API_KEY     → 用于 CI 阶段的测试

⚠️ 安全红线——Secrets 管理

  • 永远不要把 Secrets 写在 workflow 文件中——每个 GitHub Actions 日志都是公开可读的（除非使用私有仓库）
  • SSH 私钥不要设置密码（passphrase）：GitHub Actions 非交互运行，无法输入密码。确保生成私钥时直接回车（空密码）
  • 定期轮换 Secrets：如果成员离职或怀疑泄露，立即更新 ACR 密码和服务器 SSH 密钥
  • 限制服务器 SSH 来源：在安全组中，22 端口来源仅允许 GitHub Actions 的 IP 范围（GitHub Actions IP 列表），或使用跳板机方案

5.3 添加自动化测试（测试是 CI 的基石）

没有测试的 CI/CD 等于"自动部署未经检验的代码到生产"——比手动部署更可怕。为 RAG API 编写基础测试：

# tests/test_rag_api.py — RAG API 的基础测试
# 目录结构：rag-service/tests/test_rag_api.py

import pytest
from fastapi.testclient import TestClient
from app.main import app  # 从你的 FastAPI 应用导入


client = TestClient(app)


class TestHealth:
    """健康检查端点测试"""

    def test_health_endpoint(self):
        """健康检查应该返回 200"""
        resp = client.get("/health")

第六部分：蓝绿部署与回滚策略

上面的 CI/CD 有一个隐患：当你执行 docker compose up -d 时，服务会短暂重启——在重启的几秒钟内，正在处理的请求会失败。这就是停机部署。

怎么解决？蓝绿部署。

6.1 蓝绿部署原理

核心思想：同时运行两套环境——"蓝"（当前生产）和"绿"（新版本）。切换时只改 Nginx 指向的目标，而不是停掉服务重启。

正常状态：
用户 → Nginx → 蓝环境 (v1.0) ← 当前正式版本
              绿环境 (v1.0) ← 闲置

部署新版本：
用户 → Nginx → 蓝环境 (v1.0) ← 仍然服务
              绿环境 (v1.1) ← 部署新版本

切换：
用户 → Nginx → 绿环境 (v1.1) ✅
              蓝环境 (v1.0) ← 保留，准备回滚

回滚（如果需要）：
用户 → Nginx → 蓝环境 (v1.0) ← 一秒切换回来

关键是：任何时候都有一整套环境在运行，切换是瞬时的，不涉及容器的启动/停止过程。

6.2 Docker Compose 蓝绿部署实现

不需要 Kubernetes——在 Docker Compose 中通过 Compose Profiles 和 Nginx 配置切换也能实现蓝绿部署。

第一步：创建两份 docker-compose 文件

# docker-compose.blue.yml — 蓝环境（当前生产）
services:
  rag-api-blue:
    image: ${IMAGE_TAG:-registry.xxx.com/ns/rag-service:latest}
    container_name: rag-api-blue
    restart: unless-stopped
    expose:
      - "8000"
    environment:
      - LLM_PROVIDER=${LLM_PROVIDER:-openai}
      - LLM_MODEL=${LLM_MODEL:-gpt-4o-mini}
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      # ... 其他环境变量
    volumes:
      - chroma_data_blue:/app/data/chroma
    depends_on:
      chroma:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - rag-network

# docker-compose.green.yml — 绿环境（新版本）
# 结构和 blue 完全一样，只是名称改为 rag-api-green

第二步：Nginx 配置——通过上游组切换

# 两个 upstream 组——Nginx 加上变量控制切换
upstream rag_api_blue {
    server rag-api-blue:8000;
}

upstream rag_api_green {
    server rag-api-green:8000;
}

server {
    listen 443 ssl http2;
    # ... SSL 配置同前 ...

    location /api/ {
        # 通过变量 $upstream_group 动态切换（默认指向蓝环境）
        # 这个变量由 Nginx 的 map 指令或通过 nginx -s reload 切换配置
        proxy_pass http://rag_api_blue;
    }
}

第三步：部署脚本——一键切换

#!/bin/bash
# scripts/deploy-bluegreen.sh — 蓝绿部署脚本
#
# 用法：
#   ./deploy-bluegreen.sh deploy    # 部署新版本到非活跃环境并切换
#   ./deploy-bluegreen.sh rollback  # 回滚到上一个版本
#   ./deploy-bluegreen.sh status    # 查看当前活跃环境

set -e

REGISTRY="registry.cn-hangzhou.aliyuncs.com"
NAMESPACE="your-namespace"
IMAGE_NAME="rag-service"
IMAGE_TAG="${REGISTRY}/${NAMESPACE}/${IMAGE_NAME}:latest"

# 读取当前活跃环境（存储在文件中的状态）
STATE_FILE="/opt/rag-service/.deploy-state"
if [ -f "$STATE_FILE" ]; then
    ACTIVE=$(cat "$STATE_FILE")
else
    ACTIVE="blue"
    echo "$ACTIVE" > "$STATE_FILE"
fi

if [ "$ACTIVE" = "blue" ]; then
    INACTIVE="green"
    INACTIVE_UPSTREAM="rag_api_green"
    ACTIVE_UPSTREAM="rag_api_blue"
else
    INACTIVE="blue"
    INACTIVE_UPSTREAM="rag_api_blue"
    ACTIVE_UPSTREAM="rag_api_green"
fi

deploy() {
    echo "🌍 当前活跃：${ACTIVE}"
    echo "🔄 部署新版本到：${INACTIVE}"

    # 1. 拉取最新镜像
    docker pull ${IMAGE_TAG}

    # 2. 启动非活跃环境（使用新的镜像）
    export IMAGE_TAG
    docker compose -f docker-compose.${INACTIVE}.yml up -d --build

    # 3. 等待健康检查
    echo "⏳ 等待 ${INACTIVE} 环境就绪..."
    for i in $(seq 1 12); do
        if docker ps --filter "name=rag-api-${INACTIVE}" --format "{{.Status}}" | grep -q "healthy"; then
            echo "✅ ${INACTIVE} 环境健康"
            break
        fi
        if [ $i -eq 12 ]; then
            echo "❌ ${INACTIVE} 环境未就绪，回滚"
            docker compose -f docker-compose.${INACTIVE}.yml down
            exit 1
        fi
        echo "  ...等待... (${i}/12)"
        sleep 5
    done

    # 4. 切换 Nginx 指向非活跃环境 = 切换流量
    echo "🔄 切换流量到 ${INACTIVE}..."
    # 用 sed 替换 Nginx 配置中的 proxy_pass 目标
    sed -i "s|proxy_pass http://${ACTIVE_UPSTREAM}|proxy_pass http://${INACTIVE_UPSTREAM}|" /opt/rag-service/nginx.conf
    docker exec rag-nginx nginx -s reload

    # 5. 更新状态
    echo "${INACTIVE}" > "$STATE_FILE"
    echo "✅ 当前活跃：${INACTIVE}"

    # 6. 保留旧环境 5 分钟（用于快速回滚），然后停止
    # （实际中可保留更久，或通过任务定时清理）
    # docker compose -f docker-compose.${ACTIVE}.yml down
    echo "💡 旧环境 (${ACTIVE}) 保留中，5 分钟内可快速回滚"
}

rollback() {
    echo "↩️  回滚到上一个版本（${ACTIVE} → ${ACTIVE} 的重部署）..."
    # 重新部署旧版本
    # 如果是手工回滚到明确的旧版本，需要指定旧版本的 IMAGE_TAG
    echo "请手动执行：docker compose -f docker-compose.${ACTIVE}.yml up -d"
}

status() {
    echo "当前活跃环境：${ACTIVE}"
    echo "非活跃环境：${INACTIVE}"
    docker ps --format "table {{.Names}}\t{{.Status}}" | grep rag-api
}

case "${1:-status}" in
    deploy)   deploy ;;
    rollback) rollback ;;
    status)   status ;;
    *)        echo "用法: $0 {deploy|rollback|status}" ;;
esac

蓝绿部署的注意事项

  • 资源翻倍：蓝绿部署同时运行两套 RAG API 实例，内存占用翻倍。如果服务器是 8GB 内存，两套 RAG API + Chroma + Redis + Nginx 可能接近 6-7GB——确保内存足够
  • 共享基础设施：Chroma 和 Redis 可以共享（只切换 API 层），不需要两套数据库——数据库通常是状态最多的部分
  • 兼容数据库 schema：如果新版本改了 Chroma 的 collection schema，旧版本仍然需要能兼容——否则回滚后服务不可用
  • 简化版（非蓝绿）也够用：如果你的团队规模小、对停机容忍度高（维护窗口夜间操作），蓝绿部署不是必需的。简单的 docker compose up -d 重启中间停机几秒钟，很多场景可接受

# 替换原有的 _search_vector_db 方法

def _search_vector_db(self, query: str, top_k: int,
                      score_threshold: float) -> list[dict]:
    """使用 Milvus 托管服务检索"""
    from pymilvus import connections, Collection, utility

    # 连接到 Milvus 托管服务
    connections.connect(
        alias="default",
        host=settings.milvus_host,
        port=settings.milvus_port,
        user=settings.milvus_user,
        password=settings.milvus_password,
    )

    collection = Collection(settings.milvus_collection)
    collection.load()

    embedding = self._get_embedding(query)

    search_params = {
        "metric_type": "COSINE",
        "params": {"nprobe": 10},
    }

    results = collection.search(
        data=[embedding],
        anns_field="vector",
        param=search_params,
        limit=top_k,
        output_fields=["text", "source", "metadata"],
    )

    items = []
    for hits in results:
        for hit in hits:
            if score_threshold > 0 and hit.score < score_threshold:
                continue
            items.append({
                "text": hit.entity.get("text"),
                "score": hit.score,
                "source": hit.entity.get("source", ""),
                "metadata": hit.entity.get("metadata", {}),
            })

    collection.release()
    connections.disconnect("default")
    return items

故障诊断：常见问题与排查

❌ 问题一：HTTPS 证书申请失败（Certbot 报错）

现象：certbot --nginx 执行时报错说无法验证域名。

  1. DNS 解析确认：从外网执行 dig your-domain.com 确认解析到你的服务器 IP；很多情况下问题出在这里——DNS 记录还没生效或者配错了
  2. 端口可达性：从外网执行 curl -I http://your-domain.com/.well-known/acme-challenge/test——如果不通，检查安全组和 UFW 是否放行了 80 端口
  3. 备案检查：国内服务器且域名未备案，Let's Encrypt 验证请求可能被运营商拦截。用 curl -I http://your-domain.com 看响应是否有异常
  4. Nginx 正在监听 80：确保端口没有被 Docker Nginx 占用。注意：如果 Docker Nginx 已经在监听 80，系统 Nginx 会冲突——先用 docker compose stop nginx 释放端口

❌ 问题二：GitHub Actions 连接到服务器的 SSH 失败

现象：Workflow 中 SSH 步骤报错 "connect to host xxx port 22: Connection refused" 或 "Host key verification failed"。

排查步骤：

  1. SSH 端口放行：安全组中 22 端口要对 GitHub Actions Runner 的 IP 放行（或至少 0.0.0.0/0 测试用）
  2. SSH 密钥验证：在本地终端手动执行 ssh -i ~/.ssh/id_ed25519 root@<SERVER_HOST> 是否能连接？如果能，确认你在 GitHub Secrets 中配置的 SERVER_SSH_KEY 与此私钥一致
  3. known_hosts 问题：GitHub Actions 环境下首次连接服务器会提示确认主机指纹——非交互环境会卡住。appleboy/ssh-action 默认跳过主机密钥检查（不安全但方便），如果需要更安全，可以设置 fingerprint: ${{ secrets.SERVER_FINGERPRINT }}
  4. 用户身份：确认 SERVER_USERNAME 有足够的权限执行 Docker 命令（通常用 root）

❌ 问题三：docker compose up 后，容器退出代码 137（OOMKilled）

现象：docker compose ps 显示容器状态 "Exited (137)"。

原因：137 = 128 + 9（SIGKILL），通常是内存不足被内核 OOM Killer 杀掉。

排查步骤：

  1. dmesg | grep -i oom——查看内核日志确认 OOM 事件
  2. free -h——查看服务器剩余内存
  3. docker stats——实时查看各容器内存占用
  4. 解决方案：
    • 如果持续 OOM → 升级服务器规格（2C8G → 4C16G）
    • 如果 Chroma 占用太多 → 减少集合中的数据量
    • 如果同时部署了两套 RAG API（蓝绿部署）→ 共享一个 Chroma/Redis，只增加 API 实例
    • 临时解决：给容器加内存限制 deploy: resources: limits: memory: 2G

❌ 问题四：部署后 /ask 返回 502，但 /health 是 healthy

现象：/health 返回 200，但 /ask 返回 502 Bad Gateway。Nginx 日志显示 "upstream timed out"。

原因：LLM 生成回答耗时超过 Nginx 的 proxy_read_timeout。

解决方案：

  1. 增大 Nginx 的 proxy_read_timeout 到 120s 或更长（特别在使用 DeepSeek 或通义千问等国内模型时，API 响应可能更慢）
  2. 检查 LLM API 的响应时间：docker compose logs rag-api | grep '"latency_ms"'——看实际耗时
  3. 如果 LLM 本身很慢（比如用了本地模型），考虑切换更快的小模型（gpt-4o-mini 代替 gpt-4o）
  4. 使用异步流式输出（stream=True），让首字延迟降低

❌ 问题五：docker login 到阿里云 ACR 失败

现象：docker login registry.cn-hangzhou.aliyuncs.com 报错 "unauthorized: authentication required"。

排查步骤：

  1. 确认用户名：阿里云 ACR 的用户名是阿里云账号的完整登录邮箱（不是子账号名）
  2. 确认密码：ACR 使用固定密码，不是阿里云登录密码。在 ACR 控制台 → 访问凭证 → 设置固定密码。注意固定密码创建后不会再次显示，丢失只能重置
  3. 实例地域：仓库地址中的 cn-hangzhou 要与你的 ACR 实例地域一致
  4. GitHub Actions 中：不需要登录交互，使用 docker/login-action@v3 传入 Secrets
