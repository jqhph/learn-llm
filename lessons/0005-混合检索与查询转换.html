向量检索的盲区

你的 RAG 系统已经能用 embedding 做语义检索了——问"退货政策是什么"能准确找到退货相关的 chunk。这很强大，但当你部署到真实客服场景时，你会发现一条让人沮丧的规律：

现实中的"长尾问题"往往不是语义问题

实际客服日志中，大量查询根本不是"理解语义"的问题，而是精确匹配的问题：

  • "订单号 SK20240615 现在什么状态？"——需要精确匹配"SK20240615"
  • "iPhone 14 Pro 的充电头是多少瓦的？"——需要精确匹配型号"iPhone 14 Pro"
  • "SD-200 和 SD-300 有什么区别？"——需要区分两个非常相似的型号
  • "我的发票代码 3701256712 查不到"——需要精确匹配发票编号

这些查询中，embedding 模型的分词（tokenization）会把"SK20240615"切成多个子词，向量化后丢失了精确匹配信息。而 BM25 这样的关键词方法可以完美匹配——因为它不依赖语义理解，只看词频。

这里的核心洞察是：语义检索和关键词检索不是"谁替代谁"的关系，而是互补的。它们各自擅长不同的查询类型：

查询类型举例向量检索关键词检索 (BM25)
语义理解型"退货要什么条件"⭐⭐⭐⭐⭐⭐
精确匹配型"订单号 SK20240615"⭐⭐⭐⭐⭐⭐⭐
近义词型文档写"运费"用户问"邮费"⭐⭐⭐⭐⭐⭐
型号区分型"SD-200 vs SD-300"⭐⭐⭐⭐⭐⭐⭐
概括性问题"有哪些支付方式"⭐⭐⭐⭐⭐⭐⭐
罕见缩略词"WMS 入库流程"⭐⭐⭐⭐⭐⭐

理想的 RAG 系统应该同时拥有两种能力——先用两种方法各自检索，再把结果融合。这就是混合检索（Hybrid Search）。

第一步：理解 BM25——从 TF-IDF 到现代关键词检索

BM25 的核心直觉

BM25（Best Matching 25）是关键词检索的行业标准。它评估文档和查询匹配程度的方式基于三条直觉：

  1. 词频（TF）收益递减：一个词在第一篇文档中出现 10 次的匹配度，不是出现 1 次的 10 倍——而是趋近饱和（log 增长）。BM25 用参数 k1 控制这个饱和度。
  2. 逆文档频率（IDF）：在所有文档中都出现的词（如"的"、"是"）几乎没有区分度。BM25 给罕见的词更高的权重。
  3. 文档长度归一化：长的文档天然包含更多词、有更高的匹配概率。BM25 用参数 b 惩罚过长的文档，防止长文档总是排在前面。

BM25 的经典公式

Score(D, Q) = Σ [ IDF(q_i) × TF(q_i, D) × (k1 + 1) / (TF(q_i, D) + k1 × (1 - b + b × |D|/avgdl)) ]

  其中：
  - q_i        = 查询中的第 i 个词
  - TF(q_i,D)  = 该词在文档 D 中的出现次数
  - |D|        = 文档 D 的长度（词数）
  - avgdl      = 所有文档的平均长度
  - k1         = 词频饱和度控制（默认 1.2-2.0，越大越"容许多次出现"）
  - b          = 长度归一化强度（默认 0.75，0=不惩罚长文档，1=完全惩罚）

你不必记住这个公式——rank_bm25 库已经帮你实现了。但理解这三个直觉可以帮助你调参。

"""bm25_retriever.py — BM25 关键词检索模块"""
import jieba  # 中文分词
from rank_bm25 import BM25Okapi
from typing import Any
import json
import os


class BM25Retriever:
    """基于 BM25 的关键词检索器，支持中文分词。"""

    def __init__(self, k1: float = 1.5, b: float = 0.75):
        self.k1 = k1
        self.b = b
        self.bm25 = None
        self.documents: list[dict] = []   # 原始文档列表
        self.corpus: list[str] = []        # 纯文本列表（用于 BM25 索引）

    def _tokenize(self, text: str) -> list[str]:
        """中文分词，返回词列表。"""
        return list(jieba.cut(text))

    def build_index(self, documents: list[dict]) -> None:
        """
        构建 BM25 倒排索引。

        documents: [{"id": "...", "text": "...", "metadata": {...}}, ...]
        """
        self.documents = documents
        self.corpus = [doc["text"] for doc in documents]

        # 对每篇文档做分词
        tokenized_corpus = [self._tokenize(text) for text in self.corpus]

        # 构建 BM25 索引
        self.bm25 = BM25Okapi(tokenized_corpus, k1=self.k1, b=self.b)

    def retrieve(self, query: str, top_k: int = 10) -> list[dict]:
        """检索并返回 top-k 结果。"""
        if self.bm25 is None:
            raise RuntimeError("请先调用 build_index() 构建索引。")

        tokenized_query = self._tokenize(query)
        scores = self.bm25.get_scores(tokenized_query)

        # 按分数排序，取 top-k
        ranked_indices = sorted(
            range(len(scores)),
            key=lambda i: scores[i],
            reverse=True,
        )[:top_k]

        results = []
        for idx in ranked_indices:
            results.append({
                "id": self.documents[idx]["id"],
                "text": self.documents[idx]["text"],
                "score": float(scores[idx]),
                "retriever": "bm25",
            })
        return results

    def save_index(self, path: str) -> None:
        """保存索引到磁盘（保存文档列表，重建时重新分词建索引）。"""
        with open(path, "w", encoding="utf-8") as f:
            json.dump({"documents": self.documents, "k1": self.k1, "b": self.b}, f, ensure_ascii=False)

    @classmethod
    def load_index(cls, path: str) -> "BM25Retriever":
        """从磁盘加载索引。"""
        with open(path, "r", encoding="utf-8") as f:
            data = json.load(f)
        retriever = cls(k1=data["k1"], b=data["b"])
        retriever.build_index(data["documents"])
        return retriever

为什么用 jieba 分词？

BM25 是词级（token-level）的匹配算法。英文天然按空格分词，但中文需要分词器。我们用 jieba 做基础中文分词。如果是纯英文场景，直接用 .split() 即可。

注意：分词质量直接影响 BM25 的检索效果。"订单号 SK20240615"如果被切成"订单号" + "SK" + "20240615"，用户查"SK20240615"时仍然能匹配到。但如果分词把"SK20240615"当作一个整体切不出来，就需要自定义词典——企业场景中，维护 SKU/型号/地名的自定义词典是好的实践。

第二步：混合检索——用 RRF 融合两种排序

有了向量检索（第 2 课）和 BM25 关键词检索（刚才写的），现在的问题是：如何把两个独立的排序结果融合成一个统一的排序？

直接比较分数行不通——向量检索的分数（cosine similarity）范围是 [-1, 1]，而 BM25 的分数范围没有上界（可能 0-50+）。它们的分布完全不同，不能直接相加。

RRF（Reciprocal Rank Fusion）——简单的排名融合策略

RRF 的设计非常优雅：它不看分数，只看排名。基本思想是：如果一个文档在多个排序器中的排名都比较靠前，它就应该排在最终结果的前面。

RRF Score(d) = Σ( 1 / (k + rank_r(d)) )

  其中：
  - rank_r(d) = 文档 d 在第 r 个排序器中的排名（从 1 开始）
  - k         = 常数（默认 60，控制排名靠后的文档的"保底分"）

  直觉：排第 1 → 贡献 1/61 ≈ 0.0164
        排第 10 → 贡献 1/70 ≈ 0.0143
        排第 60 → 贡献 1/120 ≈ 0.0083
        差异不大
        排第 1 vs 排第 10 只差 0.0021——RFF 不"过度惩罚"排名靠后的文档

在 RRF 中，文档即使在一个排序器中排名靠后，只要在另一个排序器中排名极高，仍然有机会排到前面。这一点对混合检索至关重要——因为一个查询可能适合向量检索，也可能适合 BM25，我们不知道哪个排序器会表现更好。

"""hybrid_retriever.py — 向量 + BM25 混合检索器"""
from typing import Callable
import numpy as np


class HybridRetriever:
    """
    双路召回 + RRF 融合排序。

    使用方式：
    hybrid = HybridRetriever(vector_retriever, bm25_retriever)
    hybrid.retrieve(query)  # 自动双路召回 → RRF 融合
    """

    def __init__(
        self,
        vector_retriever: Callable | None = None,
        bm25_retriever: Callable | None = None,
        rrf_k: int = 60,
        weights: tuple[float, float] = (1.0, 1.0),
        top_k_retrieve: int = 20,
        top_k_final: int = 5,
    ):
        """
        vector_retriever: 接收 query 返回 [{"id":..., "score":...}, ...]
        bm25_retriever:   同上
        rrf_k:            RRF 常数 k（默认 60）
        weights:          两个检索器的加权系数 (w_vector, w_bm25)
        top_k_retrieve:   每个检索器召回多少条
        top_k_final:      融合后最终保留多少条
        """
        self.vector_retriever = vector_retriever
        self.bm25_retriever = bm25_retriever
        self.rrf_k = rrf_k
        self.weights = weights
        self.top_k_retrieve = top_k_retrieve
        self.top_k_final = top_k_final

    def _rrf_score(self, rank: int) -> float:
        """计算单个排名的 RRF 贡献。"""
        return 1.0 / (self.rrf_k + rank)

    def retrieve(self, query: str) -> list[dict]:
        """双路召回 → RRF 融合 → 返回 top_k_final。"""
        # 收集所有候选文档（用 dict 按 id 去重）
        candidates: dict[str, dict] = {}
        debug_info = {}

        # 1. 向量检索
        if self.vector_retriever:
            vec_results = self.vector_retriever(query, top_k=self.top_k_retrieve)
            for i, doc in enumerate(vec_results):
                doc_id = doc["id"]
                if doc_id not in candidates:
                    candidates[doc_id] = {
                        "id": doc_id,
                        "text": doc.get("text", ""),
                        "rrf_score": 0.0,
                        "vector_rank": None,
                        "bm25_rank": None,
                        "vector_score": doc.get("score", 0.0),
                        "bm25_score": None,
                    }
                candidates[doc_id]["vector_rank"] = i + 1
                candidates[doc_id]["rrf_score"] += (
                    self.weights[0] * self._rrf_score(i + 1)
                )
            debug_info["vector_count"] = len(vec_results)
            debug_info["vector_query_time"] = "ok"

        # 2. BM25 检索
        if self.bm25_retriever:
            bm25_results = self.bm25_retriever(query, top_k=self.top_k_retrieve)
            for i, doc in enumerate(bm25_results):
                doc_id = doc["id"]
                if doc_id not in candidates:
                    candidates[doc_id] = {
                        "id": doc_id,
                        "text": doc.get("text", ""),
                        "rrf_score": 0.0,
                        "vector_rank": None,
                        "bm25_rank": None,
                        "vector_score": None,
                        "bm25_score": 0.0,
                    }
                candidates[doc_id]["bm25_rank"] = i + 1
                candidates[doc_id]["bm25_score"] = doc.get("score", 0.0)
                candidates[doc_id]["rrf_score"] += (
                    self.weights[1] * self._rrf_score(i + 1)
                )
            debug_info["bm25_count"] = len(bm25_results)

        # 3. 按 RRF 分数排序取 top_k_final
        ranked = sorted(
            candidates.values(),
            key=lambda d: d["rrf_score"],
            reverse=True,
        )[:self.top_k_final]

        result = {
            "results": ranked,
            "debug": debug_info,
        }
        return result

RRF vs 其他融合方法

方法策略优点缺点
RRF基于排名的融合无需归一化分数；对异构排序器效果好；鲁棒忽视分数的绝对值差异（如第一名远优于第二名时）
Score Norm将分数归一化到 [0,1] 后加权平均保留了分数的相对差异需要了解分数分布；异常值会扭曲归一化
重排序器融合把两路 top-N 合并去重后，用 Cross-Encoder 重排序最精准最慢；需要额外模型推理

在初版实现中，使用 RRF 是最好的选择——无参数调优负担、对各种分数分布都鲁棒、且与第 4 课的重排序器完全兼容（你可以先用 RRF 融合，再用 Cross-Encoder 重排序：RRF → Re-rank → top-3）。

第三步：查询转换——在检索之前优化查询

混合检索解决了"用两种方法检索同一查询"的问题。但有时，查询本身的质量才是瓶颈——用户的原始查询可能太短、模糊、或用词不精确。在检索之前对查询进行转换（Query Transformation），可以让系统同时克服语义检索和关键词检索的局限性。

本课介绍三种实战中效果最好的查询转换技术：

🔀 多查询扩展
（Multi-Query）

用 LLM 把一条查询扩展成 3-5 条语义相似但表达不同的查询。对每条变体都做检索，然后融合结果。

效果最佳场景：用户查询非常短（1-3 个词），如"退货"→扩展出"退货政策""退货流程""退货条件"。

✏️ 查询重写
（Query Rewrite）

将用户的原始查询重写为更精确、更规范的版本。通常用于纠正含义不清的口语化表达。

效果最佳场景：口语化查询，如"我买了个手机拆了能退吗"→重写为"拆封商品的无理由退货政策是什么"。

🖼️ HyDE
（假想文档嵌入）

先用 LLM 根据查询生成一段"假想的正确答案"（Hypothetical Document），然后用这段假想文档的 embedding 去做检索，而不是用原始查询。

效果最佳场景：用户查询很模糊时，生成一个具体的假想答案，embedding 效果远好于短查询嵌入。

"""query_processor.py — 查询转换模块"""

from typing import Any
import openai
import hashlib
import json


class QueryProcessor:
    """
    查询转换处理器。提供三种策略：

    1. multi_query: 将一条查询扩展成多条变体，分别检索后融合
    2. rewrite:     将口语化查询重写为规范版本
    3. hyde:        先生成假想文档，再以假想文档为查询去检索
    """

    def __init__(
        self,
        model: str = "gpt-4o-mini",
        temperature: float = 0.0,
    ):
        self.model = model
        self.temperature = temperature

    def _call_llm(self, system: str, user: str) -> str:
        """调用 LLM 的辅助方法。"""
        response = openai.chat.completions.create(
            model=self.model,
            temperature=self.temperature,
            messages=[
                {"role": "system", "content": system},
                {"role": "user", "content": user},
            ],
        )
        return response.choices[0].message.content.strip()

    # ---- 策略 1：多查询扩展 ----

    MULTI_QUERY_SYSTEM = (
        "你是一个专注于知识库检索的 AI 助手。请将用户的原始问题改写为 "
        "3 到 5 个不同的变体，每个变体都要保留原问题的核心意图，但使用"
        "不同的措辞和句式。每行一个查询，不要编号。"
    )

    def multi_query(self, query: str, n_variants: int = 4) -> list[str]:
        """
        将一条查询扩展成多条语义相似的变体。

        返回: [原始查询 + n 条变体] 的列表
        """
        prompt = (
            f"请将以下问题改写为 {n_variants} 个不同措辞的变体。\n"
            f"每个变体一行，不要编号。\n\n"
            f"原始问题：{query}"
        )
        result = self._call_llm(self.MULTI_QUERY_SYSTEM, prompt)

        # 解析结果（每行一个变体）
        variants = [line.strip() for line in result.split("\n") if line.strip()]
        all_queries = [query] + variants[:n_variants]
        return all_queries

    def multi_query_retrieve(
        self,
        query: str,
        retrieve_fn: callable,
        top_k_per_query: int = 5,
        top_k_final: int = 10,
    ) -> list[dict]:
        """
        多查询扩展 → 各自检索 → RRF 融合。

        retrieve_fn(query) 返回 [{"id":..., "score":...}, ...]
        """
        variants = self.multi_query(query)

        # 每条变体检索
        all_results: dict[str, dict] = {}
        for i, vq in enumerate(variants):
            results = retrieve_fn(vq, top_k=top_k_per_query)
            for j, doc in enumerate(results):
                doc_id = doc["id"]
                if doc_id not in all_results:
                    all_results[doc_id] = {
                        "id": doc_id,
                        "text": doc.get("text", ""),
                        "rrf_score": 0.0,
                        "matched_variants": 0,
                    }
                # 使用 RRF 融合不同变体的结果（k=60）
                rrf_contrib = 1.0 / (60 + j + 1)
                all_results[doc_id]["rrf_score"] += rrf_contrib
                all_results[doc_id]["matched_variants"] += 1

        # 按 RRF 排序
        ranked = sorted(
            all_results.values(),
            key=lambda d: d["rrf_score"],
            reverse=True,
        )[:top_k_final]

        return ranked

    # ---- 策略 2：查询重写 ----

    REWRITE_SYSTEM = (
        "你是一个检索优化助手。请将用户的查询重写为更适合知识库检索"
        "的规范表述。要求：\n"
        "1. 保持核心意图不变\n"
        "2. 将口语化表达转为书面语\n"
        "3. 补充隐含的上下文信息\n"
        "4. 直接返回重写后的查询，不要解释"
    )

    def rewrite(self, query: str) -> str:
        """将口语化查询重写为规范版本。"""
        return self._call_llm(self.REWRITE_SYSTEM, f"原始查询：{query}")

    # ---- 策略 3：HyDE —— 假想文档嵌入 ----

    HYDE_SYSTEM = (
        "你是一个知识库问答助手。请根据用户的查询，生成一段可能存在于"
        "知识库中的"假想文档"片段。这段文字应该像知识库中的真实文档一样"
        "包含具体的政策、数据或说明，直接回答问题。不需要提"根据知识库"或"
        "假想"等词语。"
    )

    def hyde(self, query: str) -> str:
        """生成假想文档（Hypothetical Document）。"""
        return self._call_llm(self.HYDE_SYSTEM, f"问题：{query}")

    def hyde_retrieve(
        self,
        query: str,
        embed_fn: callable,
        retrieve_fn: callable,
        top_k: int = 10,
    ) -> list[dict]:
        """
        HyDE 检索流程：
        1. 用 LLM 生成假想文档
        2. 对假想文档做 embedding
        3. 用假想文档的 embedding 去检索（而不是用原始查询）

        embed_fn(text) 返回向量
        retrieve_fn(vector, top_k) 返回文档列表
        """
        hypothetical_doc = self.hyde(query)

        # 使用假想文档的 embedding 检索
        hypo_vector = embed_fn(hypothetical_doc)
        results = retrieve_fn(hypo_vector, top_k=top_k)

        # 标记哪些结果是 HyDE 产生的
        for r in results:
            r["retriever"] = "hyde"
            r["hypothetical_doc"] = hypothetical_doc[:100] + "..."

        return results

查询转换的成本与延迟

这三种工具都很有效，但每次转换都需要调用 LLM——这意味着额外的延迟和成本：

  • Multi-Query：1 次 LLM 调用 + N 次检索（N=变体数）。检索次数翻 3-5 倍，延迟和成本都明显增加。
  • Query Rewrite：1 次 LLM 调用 + 1 次检索。额外成本最低，推荐总是开启。
  • HyDE：1 次 LLM 调用 + 1 次 embedding + 1 次检索。适用于短查询或模糊查询，对长查询（用户本身描述得很清楚）没有额外收益。

建议的部署策略：Query Rewrite 默认开启（成本最低、收益稳定），对于短查询（< 5 个词）额外启用 Multi-Query，对长查询不做转换。HyDE 作为可选开关，用在已知的高模糊度查询上。

第四步：定义适配接口——统一的检索器调用

为了让评估器（来自第 4 课）能灵活切换不同的检索配置，我们需要一个统一的适配层。这一点至关重要——如果每次跑 A/B 测试都要改代码，你就不会去跑测试。

"""retrieval_adapter.py — 统一检索接口，方便评估器调用"""

from typing import Any, Callable
from bm25_retriever import BM25Retriever
from hybrid_retriever import HybridRetriever
from query_processor import QueryProcessor
from demo_v2 import chroma_collection, get_embedding  # 从第 2 课导入

import time


def build_chroma_retriever(top_k: int = 20) -> Callable:
    """返回一个纯向量检索函数（适配评估器格式）。"""
    def retrieve(query: str, top_k: int = top_k) -> list[dict]:
        query_vec = get_embedding(query)
        raw = chroma_collection.query(
            query_embeddings=[query_vec],
            n_results=top_k,
            include=["documents", "metadatas", "distances"],
        )
        results = []
        for i in range(len(raw["ids"][0])):
            results.append({
                "id": raw["ids"][0][i],
                "text": raw["documents"][0][i],
                "score": float(1 - raw["distances"][0][i]),
            })
        return results
    return retrieve


def build_bm25_retriever(bm25: BM25Retriever, top_k: int = 20) -> Callable:
    """返回一个 BM25 检索函数（适配评估器格式）。"""
    def retrieve(query: str, top_k: int = top_k) -> list[dict]:
        return bm25.retrieve(query, top_k=top_k)
    return retrieve


def build_hybrid_retriever(
    vector_fn: Callable,
    bm25_fn: Callable,
    rrf_k: int = 60,
    top_k_retrieve: int = 20,
    top_k_final: int = 10,
) -> Callable:
    """返回一个混合检索函数（适配评估器格式）。"""
    hybrid = HybridRetriever(
        vector_retriever=vector_fn,
        bm25_retriever=bm25_fn,
        rrf_k=rrf_k,
        top_k_retrieve=top_k_retrieve,
        top_k_final=top_k_final,
    )
    def retrieve(query: str, top_k: int = 0) -> list[dict]:
        # top_k 参数被忽略，用 hybrid 配置的 top_k_final
        r = hybrid.retrieve(query)
        return r["results"]
    return retrieve


def build_rewrite_retriever(
    query_processor: QueryProcessor,
    inner_retriever: Callable,
    top_k: int = 10,
) -> Callable:
    """返回一个"先重写、再检索"的函数。"""
    def retrieve(query: str, top_k: int = top_k) -> list[dict]:
        rewritten = query_processor.rewrite(query)
        print(f"    [重写] {query} → {rewritten}")
        return inner_retriever(rewritten, top_k=top_k)
    return retrieve


def run_comparison_eval(evaluator: Callable, golden_path: str = "./golden.json"):
    """
    运行四种检索配置的对比评估，打印对比表格。

    需要外部提供：
    - evaluator.evaluate(retrieve_fn, golden_path) 函数（来自第 4 课）
    - chroma_collection, get_embedding（来自第 2 课）
    - BM25Retriever（本课）
    - QueryProcessor（本课）
    """
    # 准备好每一种检索器
    vector_fn = build_chroma_retriever(top_k=20)

    # 1. 纯向量检索（基线）
    print("=" * 60)
    print("配置 A：纯向量检索（Baseline）")
    r1 = evaluator(vector_fn, golden_path)
    print_report(r1)

    # 2. 纯 BM25 检索
    bm25_fn = build_bm25_retriever(bm25, top_k=20)
    print("\n" + "=" * 60)
    print("配置 B：纯 BM25 检索")
    r2 = evaluator(bm25_fn, golden_path)
    print_report(r2)

    # 3. 混合检索（向量 + BM25 + RRF）
    hybrid_fn = build_hybrid_retriever(
        vector_fn, bm25_fn,
        rrf_k=60, top_k_retrieve=20, top_k_final=10,
    )
    print("\n" + "=" * 60)
    print("配置 C：混合检索（向量 + BM25 + RRF）")
    r3 = evaluator(hybrid_fn, golden_path)
    print_report(r3)

    # 4. 混合检索 + 查询重写
    qp = QueryProcessor()
    rewrite_hybrid_fn = build_rewrite_retriever(qp, hybrid_fn, top_k=10)
    print("\n" + "=" * 60)
    print("配置 D：混合检索 + 查询重写")
    r4 = evaluator(rewrite_hybrid_fn, golden_path)
    print_report(r4)

    # 5. 对比表格
    print("\n" + "=" * 60)
    print("📊 四种配置对比总结")
    print(f"  {'配置':<30s} {'Recall@1':>9s} {'Recall@3':>9s} {'Recall@5':>9s} {'MRR':>9s}")
    print(f"  {'-'*66}")
    for label, res in [
        ("A: 纯向量检索", r1),
        ("B: 纯 BM25", r2),
        ("C: 混合检索 (Vector+BM25)", r3),
        ("D: 混合检索 + 重写", r4),
    ]:
        s = res["summary"]
        print(f"  {label:<30s} {s.get('recall@1', 0):>9.4f} {s.get('recall@3', 0):>9.4f} {s.get('recall@5', 0):>9.4f} {s.get('mrr', 0):>9.4f}")

适配器模式的威力

build_*_retriever() 系列函数都返回 Callable[[str], list[dict]]——这个签名与第 4 课的 evaluate() 完全兼容。你可以把其中任何一个传给 evaluate()，评估器不需要任何改动。

并且适配器可以嵌套组合：build_rewrite_retriever(qp, build_hybrid_retriever(...)) 实现了"先重写查询，再做混合检索"。这种函数式组合模式让你可以用最小的代码量探索大量的配置组合。

你可能会观察到的模式

  • 纯 BM25 在精确匹配查询上碾压向量检索：当 golden.json 中包含订单号、SKU、发票号时，BM25 的 Recall@K 会远高于纯向量检索。
  • 混合检索在总体上优于单一检索：RRF 融合后，向量检索在语义查询上的优势和 BM25 在精确匹配上的优势互补，总体指标通常超过任何一种单一方法。
  • 查询重写对口语化查询有明显提升：添加查询重写后，MRR 通常提升 0.02-0.05，特别是对于包含"这个东西""能不能""多少钱"这类口语化表达的查询。
  • RRF k 值选择不太敏感：k=30 到 k=100 之间的结果通常很接近。默认 k=60 在绝大多数场景下表现良好。

生产级混合检索架构

本课讲到的是混合检索的核心逻辑。在生产环境中，混合检索的架构通常如下图：

用户查询 → Query Processor
Rewrite → Expand → 双路召回
Vector + BM25 → RRF 融合 → Cross-Encoder
重排序 → LLM 生成

这是目前工业级 RAG 系统中最为广泛采用的完整管道。各环节推荐状态：

环节推荐可选不推荐
Query Rewrite✅ 总是开启
Multi-Query✅ 短查询 (<5 词) 开启
HyDE✅ 高模糊度查询长查询场景
向量检索✅ 总是开启
BM25 检索✅ 总是开启
RRF 融合✅ 总是开启
Cross-Encoder 重排序✅ 总是开启

你现在应该记住的五句话

  1. 向量检索和 BM25 关键词检索是互补的，不是替代的。语义检索擅长理解意图，BM25 擅长精确匹配——两者结合才能覆盖真实世界的查询多样性。
  2. RRF 是融合两种异构排序器的最优方法。不看分数只看排名，无需归一化，k=60 是接近通用的保守默认值。
  3. 查询转换是检索前的"低垂果实"。Query Rewrite 默认开启，成本最低收益最稳定；Multi-Query 用于短查询；HyDE 用于模糊查询。
  4. 适配器模式让你能自由组合检索策略。用统一的 Callable[[str], list[dict]] 接口包装每一种检索器，评估器不用改一行代码。
  5. 用评估器驱动你的调优决策。对比实验不是要找到"最佳配置"，而是要理解你的数据在哪些查询上、什么条件下表现好或差。
