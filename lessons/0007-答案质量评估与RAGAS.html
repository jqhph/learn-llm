答案质量评估与 RAGAS：从"检得准"到"答得好"

本课目标：你已能用 Recall@K 证明检索质量。但好的检索是否必然带来好的答案？本课教你用 Faithfulness、Answer Relevancy 等指标量化答案质量，建立 RAG 系统的"最终质检关"。

这是你的 RAG 从 "检得准" 到 "答得好" 的关键一跃——也是你判断系统是否真正可用的最终标准。

好的检索 ≠ 好的答案

经过前两课的学习，你的检索系统已经很强了——Recall@5 可能超过 0.85，MRR 也处于不错水平。但让我们直面一个残酷的事实：检索质量高是答案质量高的必要非充分条件。

场景一：检索满分，答案跑题

查询："7 天无理由退货的运费谁承担？"

检索到的 chunk（最相关的）："自客户签收商品次日起 7 天内可申请无理由退货。退回商品需保持完好、不影响二次销售。运费由客户承担。定制商品不适用于无理由退货。"

LLM 回答："根据政策，7 天无理由退货需要商品保持完好、不影响二次销售。定制商品不适用于无理由退货。"

✅ Recall@5 = 1.0（完美命中），❌ 但答案根本没有回答"运费谁承担"！

场景二：幻觉——答案编造了上下文没有的信息

查询："退货的退款要多久到账？"

检索到的 chunk："退货申请审核通过后，我们会在 3-5 个工作日内处理退款。"

LLM 回答："退款通常在 1-2 个工作日内到账，具体取决于您的发卡行。"

❌ 上下文明确说 "3-5 个工作日"，LLM 却回答 "1-2 个工作日"——这是典型的忠实度失败。

这些问题，Recall@K 和 MRR 完全检测不出来——因为它们只评估检索，不评估生成。你需要一套新的指标来回答：

  1. Faithfulness（忠实度）：LLM 有没有"看到"上下文的内容？
  2. Answer Relevancy（答案相关度）：答案是不是真的在回答用户问题？
  3. Context Precision（上下文精确度）：检索返回的 chunk 中有没有太多噪音？
  4. Context Recall（上下文召回率）：检索到的 chunk 是否包含了回答所需的全部信息？

这四个指标合在一起，就是 RAGAS（Retrieval Augmented Generation Assessment）框架的核心。本课你将从零实现它们。

RAGAS 的四个指标详解

🟦 Faithfulness（忠实度）

答案的内容是否全部有上下文支撑？

方法：将答案拆成 claims → 逐一判断是否被 context 支持 → 比例

🟪 Answer Relevancy（答案相关度）

答案是否回答了用户的问题？

方法：根据答案生成伪问题 → 与原始问题计算语义相似度

🟧 Context Precision（上下文精确度）

检索返回的 chunk 中有多少是真正相关的？

方法：逐条判断 chunk 与问题的相关性 → 加权排序

🟩 Context Recall（上下文召回率）

回答所需的全部信息是否都在上下文中了？

方法：根据答案反向要求 → 判断每条要求是否被 context 覆盖

指标一：Faithfulness（忠实度）—— 最重要的答案质量指标

Faithfulness 回答的问题是："LLM 的答案中，有多少陈述是上下文直接支持的？" 这是 RAG 质量的红线指标——幻觉可以杀死任何一个 RAG 产品的信任度。

计算过程分为三步：

  1. 声明抽取（Claim Extraction）：将 LLM 的答案拆分为原子级的独立陈述。例如"运费由客户承担。定制商品不适用。" → ["运费由客户承担", "定制商品不适用"]
  2. 逐条验证（Verdict）：对每条声明，交给 LLM 判断"上下文是否支持这条声明？" → 返回 YES / NO / PARTIAL
  3. 计算比例：Faithfulness = YES 的数量 / (YES + NO + PARTIAL) 的数量

Faithfulness = 支持的声明数 / 总声明数

指标二：Answer Relevancy（答案相关度）

Answer Relevancy 回答："这个答案是不是在回答用户问的东西？" 一个 d perfect faithful 但答非所问的答案（比如复述政策原文但不回答具体问题），Faithfulness 可能很高但 Answer Relevancy 很低。

RAGAS 论文中的计算方式：

  1. 让 LLM 根据答案反向生成 N 个伪问题（如 4 个）
  2. 计算每个伪问题与原始问题的 余弦相似度（通过 embedding）
  3. 取平均相似度作为 Answer Relevancy

但实践中，更简单的方法也常用：直接让 LLM 对"答案和问题的匹配程度"做 1-5 分评分。

指标三：Context Precision（上下文精确度）

Context Precision 回答："检索返回的那些 chunk 里，是不是混了很多不相关的东西？" 有时候你的检索可能返回了 5 条中有 3 条相关、2 条噪音——噪音本身虽然不会直接破坏答案（因为 LLM 可能忽略它们），但它会占用 LLM 的上下文窗口，且增加了 LLM"被噪音误导"的风险。

计算方法：对每条返回的 chunk，按排序顺序判断其是否与问题相关，然后使用 加权精度公式，给予排在更前面的相关 chunk 更高权重。

指标四：Context Recall（上下文召回率）

Context Recall 回答："要回答这个问题，上下文中缺不缺关键信息？" 有时候检索可能漏掉了某个重要段落——Recall@K 已经告诉你"正确答案的 chunk 没在 top-K 里"，但 Context Recall 更进一步：即使 chunk 都到了，也可能缺了回答所需的关键信息。

计算方式：将答案中做出的"声明"与上下文匹配——如果答案中提到了"运费是 15 元"，但上下文只说了"运费按距离计算"而没有具体金额，那这条声明就是不 recall (未召回) 的。

四个指标的关系速览

指标评估对象回答的问题重要性
Faithfulness生成答案 vs 上下文答案是否忠实于上下文？⭐⭐⭐⭐⭐（红线）
Answer Relevancy生成答案 vs 用户查询答案是否回答了问题？⭐⭐⭐⭐
Context Precision检索上下文 vs 用户查询检索到的 chunk 中有多少是相关的？⭐⭐⭐
Context Recall检索上下文 vs 生成答案回答所需的信息是否都在上下文中了？⭐⭐⭐

排序并不代表 Context Precision/Recall 不重要——只是从"系统故障优先级"角度，Faithfulness 问题是最致命的。如果你的系统 Faithfulness 低于 0.7，先修好它再谈其他。

一个重要的直觉

这四个指标分成两组：

  • 检索侧（Context Precision + Context Recall）：你的检索系统是否找到了对的内容？
  • 生成侧（Faithfulness + Answer Relevancy）：你的 LLM 是否正确使用了检索到的内容？

这意味着：如果某条查询的 Faithfulness 很低（如 0.4），但 Context Recall 很高（如 0.9），问题大概率在 LLM 的 prompt 上（没让它严格遵循上下文），而非检索上。反之亦然。

第一步：构建 answer_evaluator.py

现在我们将这四个指标实现为一个可复用的 Python 模块。核心思想是 LLM-as-a-Judge：不依赖标注数据集，而是让 LLM 来当裁判，对每条 (query, answer, context) 三元组打分。

"""
answer_evaluator.py — 答案质量评估器（LLM-as-a-Judge 方式）
指标：Faithfulness, Answer Relevancy, Context Precision, Context Recall

依赖：pip install openai
用法：python answer_evaluator.py
"""
import json
import os
import statistics
from typing import Any
from openai import OpenAI

client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))

# ---------- Helper: 统一的 LLM 调用 ----------

_judge_model = "gpt-4o-mini"  # 用快模型打分，省成本


def _judge(system: str, user: str, max_tokens: int = 512) -> str:
    """调用 LLM 作为裁判，返回 JSON 文本"""
    resp = client.chat.completions.create(
        model=_judge_model,
        temperature=0.0,
        messages=[
            {"role": "system", "content": system},
            {"role": "user", "content": user},
        ],
        max_tokens=max_tokens,
    )
    text = resp.choices[0].message.content.strip()
    # 尝试提取 JSON（LLM 有时会加 markdown 包裹）
    if text.startswith("```"):
        text = text.split("\n", 1)[1].rsplit("```", 1)[0].strip()
    return text


# ========== 1. Faithfulness（忠实度） ==========

_FAITHFULNESS_SYSTEM = """你是 RAG 系统的忠实度评估员。
你的任务：将给定的答案拆解为独立的、原子性的陈述（claims），然后逐一判断上下文是否支持该陈述。

输出 JSON 格式：
{
  "claims": [
    {"claim": "陈述原文", "verdict": "YES" | "NO" | "PARTIAL", "evidence": "上下文中支持/反对的原文片段"}
  ],
  "faithfulness_score": 0.0-1.0,
  "notes": "说明"
}

评分规则：
- 如果上下文中明确提到了该陈述 → YES
- 如果上下文中明确否定了该陈述 → NO
- 如果上下文中部分提到了但不够完整 → PARTIAL（例如上下文说"3-5 个工作日"，答案说"大约一周"）
- YES 计 1 分，PARTIAL 计 0.5 分，NO 计 0 分
- faithfulness_score = 总分 / claim 总数"""


def faithfulness(query: str, answer: str, contexts: list[str]) -> dict:
    """评估答案对上下文的忠实度"""
    context_text = "\n\n---\n\n".join(
        f"[文档 {i+1}] {c}" for i, c in enumerate(contexts)
    )
    user_prompt = f"""【用户查询】
{query}

【检索到的上下文】
{context_text}

【LLM 生成的答案】
{answer}"""

    result_text = _judge(_FAITHFULNESS_SYSTEM, user_prompt)
    try:
        result = json.loads(result_text)
    except json.JSONDecodeError:
        return {
            "faithfulness_score": None,
            "claims": [],
            "notes": f"JSON 解析失败。原始输出：{result_text[:200]}",
        }
    result["query"] = query
    result["answer"] = answer
    return result


# ========== 2. Answer Relevancy（答案相关度） ==========

_ANSWER_RELEVANCY_SYSTEM = """你是 RAG 系统的答案相关度评估员。
你的任务：评估 LLM 生成的答案是否真正回答了用户查询。

输出 JSON 格式：
{
  "relevancy_score": 0-5,
  "reasoning": "为什么给这个分数",
  "missing_points": ["答案中遗漏的关键点列表"],
  "irrelevant_parts": ["答案中多余/不相关的内容列表"]
}

评分标准：
5 = 答案完美精准地回答了用户查询的全部内容
4 = 答案回答了核心问题，但遗漏了次要细节
3 = 答案部分回答了问题，但关键信息缺失
2 = 答案与问题相关但方向和核心答得不准确
1 = 答案与问题勉强有关但没有直接回答
0 = 答案与问题完全无关"""


def answer_relevancy(query: str, answer: str) -> dict:
    """评估答案是否回答了用户问题"""
    user_prompt = f"""【用户查询】
{query}

【LLM 生成的答案】
{answer}"""

    result_text = _judge(_ANSWER_RELEVANCY_SYSTEM, user_prompt)
    try:
        result = json.loads(result_text)
    except json.JSONDecodeError:
        return {
            "relevancy_score": None,
            "reasoning": f"JSON 解析失败：{result_text[:200]}",
        }
    result["query"] = query
    result["answer"] = answer
    return result


# ========== 3. Context Precision（上下文精确度） ==========

_CONTEXT_PRECISION_SYSTEM = """你是 RAG 系统的上下文精确度评估员。
你的任务：对检索返回的每个上下文文档逐一判断是否与用户查询相关。

输出 JSON 格式：
{
  "documents": [
    {
      "doc_index": 1,
      "relevant": true | false,
      "reason": "判断理由"
    },
    ...
  ],
  "context_precision_score": 0.0-1.0,
  "notes": "评估结论"
}

相关定义：如果文档包含回答用户查询所需要的任何信息，则为相关。
注意精确度对排序敏感——排在更前面的相关文档应该被给予更高的重要性。"""


def context_precision(query: str, contexts: list[str]) -> dict:
    """评估检索返回的上下文中有多少是相关的"""
    context_text = "\n\n".join(
        f"[文档 {i+1}] {c}" for i, c in enumerate(contexts)
    )
    user_prompt = f"""【用户查询】
{query}

【检索返回的上下文文档列表】
{context_text}

请逐一判断每个文档是否与查询相关。"""

    result_text = _judge(_CONTEXT_PRECISION_SYSTEM, user_prompt)
    try:
        raw = json.loads(result_text)
    except json.JSONDecodeError:
        return {
            "context_precision_score": None,
            "documents": [],
            "notes": f"JSON 解析失败：{result_text[:200]}",
        }

    # 计算加权精度：相关文档按其排序位置加权
    docs = raw.get("documents", [])
    if docs:
        total_weight = sum(1.0 / (i + 1) for i in range(len(docs)))  # 总权重（假设全部相关）
        relevant_weight = sum(
            (1.0 / (d["doc_index"])) if d.get("relevant") else 0.0
            for d in docs
        )
        score = relevant_weight / total_weight if total_weight > 0 else 0.0
    else:
        score = None

    raw["context_precision_score"] = round(score, 4) if score else None
    raw["query"] = query
    return raw


# ========== 4. Context Recall（上下文召回率） ==========

_CONTEXT_RECALL_SYSTEM = """你是 RAG 系统的上下文召回率评估员。
你的任务：分析 LLM 生成的答案，列出答案中做出的所有事实性声明，然后判断这些声明是否能在检索到的上下文中找到支持。

输出 JSON 格式：
{
  "claims": [
    {
      "claim": "答案中做出的声明",
      "supported": true | false,
      "supporting_evidence": "上下文中支持的原文（如果支持的话）",
      "notes": "说明"
    }
  ],
  "context_recall_score": 0.0-1.0,
  "missing_information": ["上下文中缺失的关键信息列表"],
  "notes": "总体评估"
}

context_recall_score = 被支持的声明数 / 总声明数"""


def context_recall(query: str, answer: str, contexts: list[str]) -> dict:
    """评估上下文中是否包含了回答所需的所有信息"""
    context_text = "\n\n---\n\n".join(
        f"[文档 {i+1}] {c}" for i, c in enumerate(contexts)
    )
    user_prompt = f"""【用户查询】
{query}

【检索到的上下文】
{context_text}

【LLM 生成的答案】
{answer}"""

    result_text = _judge(_CONTEXT_RECALL_SYSTEM, user_prompt)
    try:
        result = json.loads(result_text)
    except json.JSONDecodeError:
        return {
            "context_recall_score": None,
            "claims": [],
            "notes": f"JSON 解析失败：{result_text[:200]}",
        }
    result["query"] = query
    result["answer"] = answer
    return result


# ========== 统一评估器 ==========


def full_evaluate(
    query: str,
    answer: str,
    contexts: list[str],
) -> dict:
    """对一组 (query, answer, contexts) 运行全部四个评估指标"""
    return {
        "faithfulness": faithfulness(query, answer, contexts),
        "answer_relevancy": answer_relevancy(query, answer),
        "context_precision": context_precision(query, contexts),
        "context_recall": context_recall(query, answer, contexts),
    }


def batch_evaluate(
    test_cases: list[dict],
    retrieve_fn: callable,
    generate_fn: callable,
    max_contexts: int = 5,
    verbose: bool = True,
) -> dict:
    """
    批量端到端评估。

    test_cases: [{"query": "...", "expected_answer_summary": "..."}] 可选
    retrieve_fn(query) -> list[{"id": str, "text": str, "score": float}]
    generate_fn(query, contexts) -> str
    """
    results = []
    for i, tc in enumerate(test_cases):
        query = tc["query"]
        if verbose:
            print(f"[{i+1}/{len(test_cases)}] 评估: {query[:60]}...")

        retrieved = retrieve_fn(query)[:max_contexts]
        contexts = [r["text"] for r in retrieved]
        answer = generate_fn(query, contexts)
        eval_result = full_evaluate(query, answer, contexts)

        eval_result["_meta"] = {
            "query": query,
            "context_ids": [r["id"] for r in retrieved],
        }
        results.append(eval_result)

    # 汇总
    scores = {"faithfulness": [], "answer_relevancy": [], "context_precision": [], "context_recall": []}
    for r in results:
        for key in scores:
            val = r[key].get(f"{key}_score") if key in r else r.get(f"{key}_score")
            if val is not None:
                scores[key].append(val)

    summary = {}
    for key, vals in scores.items():
        vals = [v for v in vals if v is not None]
        summary[key] = {
            "mean": round(statistics.mean(vals), 4) if vals else None,
            "std": round(statistics.stdev(vals), 4) if len(vals) > 1 else None,
            "min": round(min(vals), 4) if vals else None,
            "max": round(max(vals), 4) if vals else None,
        }

    return {"summary": summary, "details": results}


def print_report(eval_result: dict) -> None:
    """打印端到端评估报告"""
    s = eval_result["summary"]
    print(f"{'='*56}")
    print(f"📊 RAG 答案质量评估报告")
    print(f"{'='*56}")
    print(f"评估样本数：{len(eval_result['details'])}")
    print()
    for metric in ["faithfulness", "answer_relevancy", "context_precision", "context_recall"]:
        m = s.get(metric, {})
        mean = m.get("mean")
        if mean is not None:
            print(f"  {metric:<20s}  {mean:.4f} (σ={m.get('std', 'N/A')})")
    print()
    print(f"{'='*56}")
    print(f"💡 指标解读（参考阈值）：")
    print(f"  faithfulness > 0.9        → 优秀，答案忠实于上下文")
    print(f"  faithfulness 0.7-0.9     → 良好，有小幅幻觉需排查")
    print(f"  faithfulness < 0.7        → ❌ 需要紧急修复（prompt 或检索）")
    print(f"  answer_relevancy > 4.0    → 答案很好地回答了用户问题")
    print(f"  answer_relevancy < 3.0    → 答案偏离了用户的问题")
    print(f"  context_precision < 0.6   → 检索噪音偏多，需要优化检索")
    print(f"  context_recall < 0.6      → 检索漏了关键信息，需增加 top-K 或改进 chunk")
    print(f"{'='*56}")

    print(f"  Faithfulness Score: {f['faithfulness_score']}")
    for c in f["claims"]:
        print(f"  - [{c['verdict']}] {c['claim'][:50]}")

    print("\n🧪 快速测试 Answer Relevancy...")
    r = answer_relevancy(q, ans)
    print(f"  Relevancy Score: {r['relevancy_score']}/5")
    print(f"  理由：{r['reasoning'][:100]}")

    print("\n🧪 快速测试 Context Precision...")
    cp = context_precision(q, ctx)
    print(f"  Precision Score: {cp['context_precision_score']}")

    print("\n🧪 快速测试 Context Recall...")
    cr = context_recall(q, ans, ctx)
    print(f"  Recall Score: {cr['context_recall_score']}")

关于 Judge Model 的选择

上面的代码使用 gpt-4o-mini 作为审判模型。这是一个很实用的平衡点：

  • gpt-4o-mini：性价比最高，适合日常评估（推荐起始选择）
  • gpt-4o：更可靠但贵 15-20 倍，适合正式发布前的全面评估
  • 本地小模型（如 Qwen2.5-7B）：免费但可靠性较低，需要仔细调 prompt

一个经过验证的模式：平时用 gpt-4o-mini 做快速迭代，关键里程碑用 gpt-4o 跑全量评估来校准。

第二步：构建端到端评估流水线

现在把你的检索器、生成器和评估器串起来——对 golden dataset 中的每条查询，执行完整的 "检索 → 生成 → 评估" 流程。

"""
eval_pipeline.py — 端到端评估流水线

用法：
  python eval_pipeline.py --golden golden.json --retriever hybrid --top-k 5
"""
import json
import argparse
from answer_evaluator import batch_evaluate, print_report


def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--golden", default="./golden.json")
    parser.add_argument("--retriever", default="hybrid", choices=["vector", "bm25", "hybrid"])
    parser.add_argument("--top-k", type=int, default=5)
    args = parser.parse_args()

    # 加载 golden 测试集
    with open(args.golden, "r", encoding="utf-8") as f:
        test_cases = json.load(f)

    # 初始化检索器（根据你第 5 课的模块）
    from retrieval_adapter import get_retriever
    retrieve_fn = get_retriever(args.retriever)

    # 初始化生成器（根据你第 2 课的 demo）
    from rag_demo import answer_query
    def generate_fn(query, contexts):
        return answer_query(query, contexts)

    # 批量评估
    result = batch_evaluate(
        test_cases,
        retrieve_fn=retrieve_fn,
        generate_fn=generate_fn,

    print_report(result)

你应该评估什么？

端到端评估最大的价值在于 AB 比较。每次你对 RAG 系统做改动后，跑一遍评估报告并对比结果：

改动类型预期影响的指标
修改 prompt（更严格的"遵循上下文"指令）🔺 Faithfulness
更换 embedding 模型🔺 Context Precision + Context Recall
增加 top-K（5→10）🔻 Context Precision（噪音增多），有可能🔺 Context Recall
增加重排序（Re-ranking）🔺 Context Precision（把相关排前面），🔺 Faithfulness
增加查询重写（Query Rewrite）🔺 Context Precision + Context Recall
更换 LLM 模型（如 gpt-4o-mini → gpt-4o）🔺 Faithfulness + Answer Relevancy

把这当作你的 RAG 调试仪表盘——每次改动只需要跑一次 python eval_pipeline.py，就知道是变好了还是变差了。

第三步：理解 LLM-as-a-Judge 的局限

LLM-as-a-Judge 很强大，但它不是银弹。理解它的局限比理解它的用法更重要——否则你可能会被漂亮的评分报告欺骗。

已知的偏差

偏差类型表现缓解方式
位置偏差 审判模型倾向于给排在"第一个"或"最后一个"的答案更高分 多次评估不同顺序后取平均；交换上下文顺序再测
自我强化偏差 审判模型更倾向于推荐与自己同族的模型生成的答案（如 GPT-4 评分给 GPT-4 生成的高分） 使用不同家族的模型做审判（如用 Claude 评估 GPT，或反之）
长度偏差 越长的答案通常被认为越"好"，即使包含很多废话 在 prompt 中明确要求扣分项："如果答案包含冗余信息请扣分"
讨好偏差 模型倾向于说"YES"以免冒犯用户 temperature=0 + 明确的评分标准 + chain-of-thought 推理

不要犯这个经典错误

新手最容易犯的错误是：用 GPT-4 给 GPT-4 的答案打分，得到一个漂亮的分数，然后觉得自己的系统已经很好了。

实际上，GPT-4 作为审判者给 GPT-4 作为生成者的答案打分，存在严重的自我强化偏差——分数可能虚高 10-20%。正确做法：

  1. 尽量使用不同的模型族（生成用 gpt-4o-mini，审判用 Claude Haiku）
  2. 定期加入"黄金标准答案"人类打分作为校准
  3. 绝对不要只依赖一次评估的结果——如果分数变动 < 0.05，基本是噪声

什么时候应该用人工评估？

LLM 评估永远无法完全替代人工评估。以下情况下你必须引入人工：

  • 上线前的最终验收：在把系统发布给真实用户之前，至少人工审核 20-30 条有代表性的测试样本
  • 发现 Faithfulness < 0.7 但看不出来：LLM 判断"不忠实"但你不理解为什么——说明存在微妙的幻觉问题
  • 领域术语评估：如果一个句子涉及专业领域的判断，LLM 可能不如领域专家可靠
  • 评估指标在调整：当你新加了评估指标需要验证其有效性时

推荐成本模型

假设你有一个 20 条查询的 golden dataset，每条评估四个指标：

  • 每次端到端评估（gpt-4o-mini）：约 ¥0.05-0.10（20×4次LLM调用）
  • 每次端到端评估（gpt-4o）：约 ¥1.0-2.0
  • 每日运行成本：开发期每天跑 5-10 次，月成本约 ¥30-50（gpt-4o-mini 方案）

这是整个 RAG 系统优化中回报率最高的成本——没有之一。没有评估，你所有优化都是盲目的。

生产环境中的答案质量评估

当你的 RAG 系统上线后，离线评估（在 golden dataset 上跑）是不够的。真正的用户流量会暴露你在 golden dataset 中覆盖不到的问题。以下是一些生产环境实践：

在线评估 vs 离线评估

维度离线评估（本课所教）在线评估
数据人工标注的 golden dataset（20-50 条）真实用户查询（可能成千上万条）
指标Faithfulness, Relevancy, Precision, Recall用户满意度、点赞/踩率、对话轮次、放弃率
频率每次代码/配置变动后持续实时监控
成本中（每个查询 4 次 LLM 调用）高（需要采样，不能对所有用户查询跑）
主要用途开发期调试、发布前验证监控退化、发现 edge cases

生产环境策略

  • 采样评估：对所有用户请求采样 1-5%，异步运行 LLM-as-a-Judge 评估，记录到日志
  • 退化告警：设置 Faithfulness 基线（如 0.85），如果滑动窗口平均值（如最近 1000 条）低于基线则触发告警
  • A/B 测试：新版本上线时，将部分用户流量导入新版，对比两版的评估指标
  • 人工抽检：每周从低分样本（Faithfulness < 0.6）中随机抽取 10 条，人工确认是否是误判
  • 持续更新 golden dataset：从真实用户反馈中挑出典型的高质量查询，每两周补充到 golden dataset

小团队的最佳实践

如果你是独立开发者或小团队：

  1. 20 条 golden 就够了——比没有强 100 倍。不需要追求完美覆盖。
  2. 优先修 Faithfulness——一个很少但从不幻觉的系统，比一个经常幻觉偶尔答对的系统更受欢迎。
  3. 每次改动跑一遍评估——养成习惯，比任何代码审查都管用。
  4. 用 gpt-4o-mini 做审判——成本够低可以频繁跑。
  5. 两周一次人工抽检——抽 10 条看你的 LLM 审判和你的判断是否一致。

RAG 系统的质量，归根结底是"你有多频繁地在测量它"。测量驱动优化。
