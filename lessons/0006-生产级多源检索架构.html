Lesson 0006

生产级多源检索架构：为什么订单号不该进向量库

本课目标：理解为什么"把所有数据塞进向量库"不是生产级方案，掌握四种生产级多源检索架构模式，学会在真实的客服系统中正确路由查询到相应的数据源。

课程 0005 没有告诉你的

回顾课程 0005，我们讲混合检索时用了这样的例子：

"订单号 SK20240615 现在什么状态？" → BM25 精确匹配 SK20240615

"发票代码 3701256712 查不到" → BM25 精确匹配 3701256712

这些例子的前提条件是：订单号和发票代码已经在知识库的文本中存在。教程里确实是这样写的：

{"id": "sku-0", "text": "SK20240615 订单目前状态为：已发货，预计 3-5 个工作日送达。"}
{"id": "invoice-0", "text": "发票代码 3701256712 的发票已开具，可登录官网下载。"}

但你立刻发现了问题：现实中，订单号和SKU都存储在MySQL里，可能有上千万甚至上亿条记录。

—— "难道都得切块向量化存进向量库？"

答案：绝对不能。

把订单数据塞进向量库的后果

  • 成本爆炸：1 亿条订单，每条 embedding 约 1536 维 float32 = 6KB，仅向量存储就需要 ~60 GB，还不算原始文本。embedding API 调用费用另计。
  • 实时性灾难：订单在持续写入、状态变化。每次新订单都要跑 embedding→写入→建立 BM25 索引。BM25 是内存索引，重构建开销极大。
  • 检索精度灾难：搜索"SK20240615"时，向量检索会把"SK"和"20240615"切成子词，可能返回"SK20240616"甚至"2000 年 SK 公司成立"——语义相似但完全无关。
  • 安全灾难：订单包含用户隐私（姓名、地址、电话），不能放进向量库——后者缺乏列级权限控制和细粒度审计。

课程 0005 的 BM25 例子在教学层面是对的——它讲的是关键词检索的机制。但在架构层面，你问出了所有 RAG 教程都回避的关键问题：事务型数据到底该怎么接入 RAG 系统？

回顾：第 5 课：混合检索与查询转换。本课也直接关联 第 7 课——如果不能从正确的源拿到数据，评估指标再好看也没有用。

核心脑图：数据分水岭

在开始架构设计之前，你首先要理解企业数据的两大类——它们在接入 RAG 系统的方式完全不同：

数据类型例子数据量级变动频率接入方案
知识型 / 参考型售后政策、产品说明、操作手册、FAQ、法律法规千 ~ 百万级低（周/月更新）✅ 向量库 + BM25
事务型 / 记录型订单、SKU 库存、物流轨迹、用户账户、发票记录百万 ~ 亿级极高（秒/分钟级变更）❌ 不应入向量库

一个简单的判断标准

问自己："用户查这条数据，是为了读它的内容，还是为了查它的状态？"

  • 读内容：退货政策是什么？Wi-Fi 模块的规格？→ 适合向量库（文本相似度搜索）
  • 查状态：我的订单到哪了？这个 SKU 还有库存吗？→ 不适合向量库（需要精确查询+实时数据）

也就是说，课程 0005 的 BM25 例子并非"错误"，而是"放错了场景"。真正的生产架构是：把查询路由到正确的数据源，而不是把所有数据揉进一个索引。

生产级四种架构模式

以下是企业级 RAG 客服系统中最常用的四种模式，它们按复杂度从低到高排列。前两种你很快可以落地，后两种在数据源增多时自然会演化出来。

模式一：查询路由（Intent Routing）

用户查询 → 路由分类器 → 知识库检索 → LLM 回答
用户查询 → 路由分类器 → 查询订单 API / DB → 直接返回

这不是什么新技术——它在客服行业存在了几十年。在现代 RAG 中，我们只是用一个分类模型（传统分类器或 LLM）来判断用户的意图，然后路由到不同的处理管道。

最朴素的实现——用 LLM 做分类：

"""query_router.py — 最简单的意图路由"""
import openai
import json

class QueryRouter:
    """
    根据用户查询的意图，路由到对应的处理模块。

    支持三种路由目标：
    - "kb":   知识库问题 → 走向量检索 (RAG)
    - "order": 订单查询 → 查订单数据库
    - "sku":   SKU/产品查询 → 查产品数据库
    """

    ROUTING_PROMPT = """判断用户查询属于哪一类，只输出一个词：
- kb:    知识、政策、说明类问题（如退货政策、如何使用、功能介绍）
- order: 订单相关（含订单号、物流、发票）
- sku:   商品/产品相关（含SKU、型号、规格参数）
- other: 其他

用户查询：{query}

分类结果："""

    def route(self, query: str) -> str:
        """返回 'kb' | 'order' | 'sku' | 'other'"""
        prompt = self.ROUTING_PROMPT.format(query=query)
        response = openai.chat.completions.create(
            model="gpt-4o-mini",
            temperature=0.0,
            max_tokens=10,
            messages=[{"role": "user", "content": prompt}],
        )
        intent = response.choices[0].message.content.strip().lower()
        return intent if intent in ("kb", "order", "sku") else "other"

    def handle(self, query: str):
        """路由并调用对应的处理器"""
        intent = self.route(query)

        if intent == "kb":
            # → 走标准 RAG：向量检索 + LLM 生成
            return self._handle_kb(query)
        elif intent == "order":
            # → 调用订单服务 API / 查 MySQL
            order_no = self._extract_order_no(query)
            return self._query_order(order_no)
        elif intent == "sku":
            # → 调用产品服务 API / 查 MySQL
            sku_code = self._extract_sku(query)
            return self._query_product(sku_code)
        else:
            return self._handle_fallback(query)

为什么这么"简单"就够用了？

因为意图分类是一个极其简单的任务——比总结文档或写代码简单得多。即使只用一个 8B 的开源模型（如 Qwen2.5-7B）也能做到 >95% 的准确率。一些企业甚至使用基于关键词的模式匹配（规则引擎）做初筛，只对不确定的查询才调用 LLM。

注意：路由分类器只做分类，不回答具体业务问题。这意味着你可以为它选择一个便宜、快速的模型——甚至可以在边缘端部署。

生产实践：订单号的提取策略

import re

def extract_order_no(query: str) -> str | None:
    """
    从自然语言中提取订单号。

    常见订单号格式：
    - SK20240615, ORD20240615-001
    - 纯数字（如 3701256712）
    - 带前缀的混合编码
    """
    # 策略1：正则匹配已知格式
    patterns = [
        r'SK\d{6,}',           # SK 开头 + 6位以上数字
        r'ORD\d{6,}',          # ORD 开头
        r'\b\d{10,}\b',        # 10位以上纯数字
    ]
    for pat in patterns:
        match = re.search(pat, query)
        if match:
            return match.group()

    # 策略2：用 LLM 提取（处理不规则格式）
    # 仅在正则匹配不到时调用
    return None

💡 关键实践：正则优先，LLM兜底

很多 RAG 教程让人什么都用 LLM，但在生产环境中，对于"从句子中提取订单号/SKU"这种有明确结构模式的子任务，正则表达式比 LLM 更快、更稳定、更便宜（延迟 < 1ms vs 200-500ms，成本 0 vs 有成本）。

推荐策略：先正则匹配→匹配不到再用 LLM→还不行则走兜底。这叫"防御性分层调用"。

模式二：LLM Function Calling / Tool Use

这是目前工业界最主流的做法——不是让 LLM "知道"数据，而是给 LLM 查询数据的能力。

用户查询 → LLM 推理 → 调用 tool:
query_order() → LLM 生成回答

def function_calling_rag(query: str) -> str:
    """整套流程：LLM 决定调用什么 tool → 执行 → LLM 生成回答"""
    import openai

    messages = [
        {"role": "system", "content": "你是一个客服助手。使用工具获取所需数据。"},
        {"role": "user", "content": query},
    ]

    # 第一轮：LLM 决定是否调用工具
    response = openai.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
        tools=tools,
        tool_choice="auto",
    )

    msg = response.choices[0].message

    # 如果没有工具调用请求，直接返回
    if not msg.tool_calls:
        return msg.content

    # 处理每个工具调用
    for tool_call in msg.tool_calls:
        fn_name = tool_call.function.name
        args = json.loads(tool_call.function.arguments)

        if fn_name == "query_order":
            result = query_order_from_db(args["order_no"])
        elif fn_name == "get_product_by_sku":
            result = get_product_from_db(args["sku"])
        elif fn_name == "search_knowledge_base":
            result = search_vector_db(args["query"])
        else:
            result = {"error": "Unknown tool"}

        # 把工具结果返回给 LLM
        messages.append(msg)
        messages.append({
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": json.dumps(result, ensure_ascii=False),
        })

    # 第二轮：LLM 根据工具结果生成回答
    final = openai.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
    )
    return final.choices[0].message.content

为什么 Function Calling 是当前最佳方案？

  • 数据不动，能力动：不需要把订单数据复制到向量库，LLM 通过调用函数直接访问源数据。
  • 权限可控：每个工具可以有独立的访问控制。客服 Agent 能查订单，但不一定能改。
  • 审计可追踪：每一次数据查询都对应一个函数调用记录，便于事后审计。
  • 保护用户隐私：姓名、电话、地址只在函数执行时使用，不会泄露到向量库的持久化存储中。

Function Calling 的陷阱

  • LLM 可能用错参数：提取订单号失败时会调用错误 key 查询。解决方案：在函数入口做参数校验，无效时返回清晰错误信息让 LLM 重试。
  • 过多的工具会让 LLM 决策困难：超过 10-15 个工具时，LLM 可能会选错工具或忘记调用。解决方案：分层路由——先分大类（订单/知识库/产品），每个大类内部再细分。
  • 工具需要有幂等设计：用户说"查我最近的订单"，LLM 可能连续调用两次。查询型工具应该是幂等的，多次调用不产生副作用。

模式三：NL2SQL（自然语言转 SQL）

当订单查询的过滤条件复杂且多变时——例如"上个月买 iPhone 14 Pro 并且用了优惠券的订单有哪些"——为每种组合写一个 Function 是不现实的。这时 NL2SQL 模式上场：

让 LLM 把自然语言转换成 SQL 查询，直接在 MySQL 上执行。

用户查询 → LLM → SQL → 执行 SQL → LLM 组织回答 
"""nl2sql.py — 自然语言转 SQL 查询"""

ORDER_DB_SCHEMA = """
表 orders (订单表):
  - id          BIGINT PRIMARY KEY  自增主键
  - order_no    VARCHAR(32) UNIQUE  订单号 (SK20240615)
  - user_id     BIGINT              用户ID
  - status      VARCHAR(16)         订单状态: pending/paid/shipped/delivered/cancelled/returned
  - total       DECIMAL(10,2)       订单总金额
  - sku_list    JSON                商品列表 [{"sku": "AP-IP14Pro-256-BK", "qty": 1, "price": 8999}]
  - coupon_id   BIGINT              使用的优惠券ID (可为NULL)
  - created_at  DATETIME            下单时间
  - updated_at  DATETIME            最后更新时间

表 skus (商品SKU表):
  - sku         VARCHAR(32) PRIMARY KEY  商品编码
  - name        VARCHAR(128)             商品名称
  - category    VARCHAR(64)              类目
  - price       DECIMAL(10,2)            当前售价
  - stock       INT                      库存
"""

NL2SQL_SYSTEM = f"""你是一个 MySQL 查询助手。将用户的问题转换为 SQL 查询。

数据库 Schema：
{ORDER_DB_SCHEMA}

规则：
1. 只输出 SQL，不要任何解释
2. 只执行 SELECT 查询
3. 如果请求不确定/不安全，输出 "UNSAFE"
4. 如果没有匹配的数据结构可以回答，输出 "UNSUPPORTED"
5. 使用 %s 占位符做参数化查询（防止 SQL 注入）
"""

def nl2sql(user_query: str) -> str:
    """自然语言 → SQL"""
    import openai

    response = openai.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0.0,
        messages=[
            {"role": "system", "content": NL2SQL_SYSTEM},
            {"role": "user", "content": user_query},
        ],
    )
    sql = response.choices[0].message.content.strip()

    # 安全检查
    sql_upper = sql.upper()
    if sql in ("UNSAFE", "UNSUPPORTED"):
        return sql
    if not sql_upper.startswith("SELECT"):
        return "UNSAFE"

    return sql


def execute_nl_query(user_query: str) -> str:
    """完整流程：NL → SQL → 执行 → LLM 组织回答"""
    sql = nl2sql(user_query)

    if sql == "UNSAFE":
        return "抱歉，我无法执行此查询。请提供更具体的信息。"
    if sql == "UNSUPPORTED":
        # 兜底：尝试用工具函数做模糊匹配
        return handle_fallback(user_query)

    # 执行 SQL（使用参数化查询）
    # import pymysql
    # conn = pymysql.connect(...)
    # with conn.cursor() as cur:
    #     cur.execute(sql)
    #     rows = cur.fetchall()
    #     columns = [desc[0] for desc in cur.description]

            {"role": "user", "content": f"用户问题：{user_query}\n查询结果：{json.dumps(rows, ensure_ascii=False, default=str)}"},
        ],
    )
    return response.choices[0].message.content

⚠️ NL2SQL 的安全红线

  • 永远限制为 SELECT 只读：在数据库连接层面，使用一个只读权限的数据库用户连接到只读副本，而不是主库。
  • SQL 注入防御：始终使用参数化查询（%s 占位符），不要让 LLM 直接拼接值到 SQL 字符串中。
  • 结果行数上限：在生成的 SQL 末尾追加 LIMIT 100，防止"查所有订单"导致数据库 OOM。
  • 执行时间限制：设置查询超时（如 5 秒），超时后终止并告诉用户查询太复杂，建议简化。
  • 敏感列脱敏：如果没有必要，Schema 描述中不要包含电话、地址等 PII 列，或者告诉模型"返回结果时脱敏"。

模式四：联邦检索 (Federated Retrieval)

当数据源超过 3-4 个时（知识库 + 订单库 + 商品库 + 库存系统 + 物流系统），一个查询可能同时需要多个数据源的信息。例如："我的 SK20240615 订单里的 iPhone 充电头是多少瓦的？"——这既需要订单信息（买了什么），又需要产品规格信息（充电头参数）。

用户查询 → 查询分解器 → 知识库检索
→ 订单查询
→ 产品搜索 (ES) → 结果融合 → LLM 综合回答 
"""federated_retriever.py — 联邦检索"""

class FederatedRetriever:
    """
    多源联邦检索：将一个查询分解成多个子查询，分别到不同数据源检索，
    最后汇总融合。
    """

    def __init__(self):
        self.sources = {
            "kb": KnowledgeBaseRetriever(),   # 向量检索
            "order": OrderServiceClient(),     # 订单服务 gRPC/HTTP
            "product": ProductSearchClient(),  # Elasticsearch
            "logistics": LogisticsAPIClient(), # 物流 API
        }

    def decompose_query(self, query: str) -> list[dict]:
        """
        将查询分解为多个子查询，每个指定数据源。

        返回: [
            {"source": "order", "params": {"order_no": "SK20240615"}},
            {"source": "product", "params": {"query": "iPhone 14 Pro 充电头 功率"}},
        ]
        """
        # 使用 LLM 分解
        prompt = f"""将以下客服查询分解为子任务，每个子任务指定数据源：

        可用数据源：
        - kb: 客服知识库（政策、流程、说明）
        - order: 订单系统（订单号、物流状态）
        - product: 产品库（SKU、规格参数、价格）
        - logistics: 物流系统（运单追踪）

        查询：{query}

        以 JSON 数组格式输出，不要解释：
        [{{"source": "...", "params": {{...}}}}, ...]
        """
        # ... 调用 LLM 解析
        return [
            {"source": "order", "params": {"order_no": "SK20240615"}},
            {"source": "product", "params": {"query": "iPhone 14 Pro 充电头"}},
        ]

    def retrieve(self, query: str) -> dict:
        """联邦检索：分解 → 并行查询各数据源 → 聚合 """
        sub_tasks = self.decompose_query(query)

        # 并行查询所有数据源
        from concurrent.futures import ThreadPoolExecutor
        results = {}

        with ThreadPoolExecutor(max_workers=4) as pool:
            futures = {}
            for task in sub_tasks:
                source = task["source"]
                params = task["params"]
                handler = self.sources.get(source)
                if handler:
                    future = pool.submit(handler.query, **params)
                    futures[future] = source

            for future in futures:
                source = futures[future]
                results[source] = future.result()

        return results

联邦检索是企业级 RAG 的终局形态。它不是"把所有数据倒入一个引擎"，而是"让无数个引擎以统一接口对外服务"。知名开源实现包括 LlamaIndex 的 RouterQueryEngine 和 LangChain 的 MultiSourceRetriever。

从你的客服场景出发：推荐演进路径

  1. 第一步：查询路由（模式一）+ 知识库向量检索（已有）。这是你现在就可以做的 —— 两三天的改动量。
  2. 第二步：将订单查询和产品查询升级为 Function Calling（模式二）。不需要重构现有路由。
  3. 第三步：当数据源超过 4 个且查询需要同时跨多个源时，考虑引入联邦检索（模式四）。

NL2SQL（模式三）在你需要灵活过滤（时间范围、条件组合）时引入，不要作为默认方案——它对数据库安全性有额外要求。

完整架构鸟瞰：企业级 RAG 客服系统

现在把一切拼起来，一个生产级的客服 RAG 系统长什么样：

用户消息 → 意图分类
LLM / 规则
├─ 知识库类 → 向量检索
Chroma/Pinecone → BM25 (ES)
备胎召回 → RRF 融合 → LLM 回答
引用知识库文档
├─ 订单类 → Function: query_order() → LLM 组织回答
调用结果 + 引用
├─ 产品类 → Elasticsearch 搜索 → LLM 组织回答
└─ 混合查询 → 查询分解器 → 多源并行取数 → LLM 综合回答

关键观察：

  • 向量库（Chroma / Pinecone / Weaviate）只存知识型数据——政策文档、产品手册、FAQ。数据量通常从几千到几百万条不等。
  • 事务型数据（订单、库存、物流）存在源数据库——MySQL, PostgreSQL, 或者 Elasticsearch（用于全文搜索）。
  • Elasticsearch 是"中间地带"——对于需要模糊搜索的产品库（"iPhone 充电头"搜出所有兼容充电头），Elasticsearch 的全文搜索能力优于向量检索，也优于精确的 SQL 查询。

你还想问的问题

"那 BM25 在真实生产中还用在什么场景？"

课程 0005 的 BM25 没有错——它只是应该用在知识库中那些包含编码/型号的文本上，而不是用来替代订单数据库。比如：

  • "AP-IP14Pro-256-BK 这个 SKU 的维修指南是什么？"——SKU 型号出现在文档文本中，BM25 完美匹配
  • "iPhone 14 Pro 的 A16 芯片有什么改进？"——"A16"是产品参数，出现在技术文档中
  • "退货单 RMA20240088 的处理流程"——退货单号存在于售后流程文档的示例中

知识库文档中本身就会提到的编码和型号才是 BM25 的用武之地。不存在于知识库中的海量事务数据（实际订单、实际库存）不走 BM25。

"为什么课程 0005 用了误导性的例子？"

这是一个合理的批评。课程 0005 的例子的确打破了"数据应该在哪"的假设。但它的真正意图是展示 BM25 的精确匹配机制——只是选了一个不太恰当的场景。本课就是对那个架构缺口的补全。

"实际客服系统里有向量库吗？"

有，但只是整个系统的一部分。根据某电商客服团队的公开分享，他们的生产系统中：

  • 向量库（~50万条文档 chunk）只覆盖了知识型查询的约 30%
  • 订单/物流查询（约 50%）直接路由到业务 API
  • 产品搜索（约 15%）通过 Elasticsearch
  • 其余 5% 走兜底/人工

向量检索是 RAG 系统中最亮眼的组件，但不一定是调用量最大的组件。

练习：设计你自己的多源检索架构

实战练习

假设你正在为一家中等规模的电商公司设计客服 RAG 系统。已有以下数据：

  • 知识库：2000 篇 Markdown 文档（售后政策、操作指南、常见问题）
  • 订单表：5000 万条订单记录，在 MySQL 中
  • 商品表：10 万条 SKU，含详细规格参数，在 MySQL 中
  • 物流轨迹：亿级，在专门的物流系统（API 接口）中
  • 用户评价：200 万条，在 MongoDB 中

任务：为以下每种用户查询，写出处理路径——它应该经过什么组件、查询什么数据源、怎么组织回答。用文字描述即可：

  1. "iPhone 14 Pro 的保修政策是什么？"
  2. "帮我查一下订单 SK20240615 到哪了"
  3. "AP-IP14Pro-256-BK 还有库存吗？"
  4. "我买的 iPhone 充电头是多少瓦的？"（提示：先查订单里的SKU，再查产品规格）

写完后发给我，我帮你分析架构设计中的问题。

选择题：巩固理解

以下哪种做法在生产环境中是正确的？

  1. 把所有订单数据（包括用户姓名、电话）都 embedding 后存入向量库，这样可以搜索订单内容
  2. 使用 LLM Function Calling，让 LLM 按需调用 query_order(order_no) 查询订单 API，订单数据不进入向量库
  3. 把 1 亿条订单全部导入 BM25 索引，用关键词搜索代替数据库查询
  4. 告诉用户"请自己到官网查订单"，避免麻烦的架构设计

答案：B。订单数据是事务型隐私数据，不应持久化进入向量库。Function Calling 让 LLM 按需调用 API 去源数据库查询，是最安全且优雅的解决方案。

你现在应该记住的五句话

  1. 数据分两类：知识型放向量库，事务型留源数据库。前者查询语义相似度，后者查询精确状态——它们是不同的技术栈。
  2. 查询路由是 RAG 系统的第一道门。先判断用户意图，再路由到对应的数据源——90% 的场景下这是最高效的做法。
  3. LLM Function Calling 是目前连接 LLM 和事务数据的工业级标准。给 LLM 调用数据库的"能力"，而不是把数据复制一份给它。
  4. 安全性第一：NL2SQL 必须用只读用户 + 参数化查询 + 结果行限 + 超时。让 LLM 生成 SQL 去操作生产数据库是一个需要严肃对待的安全决策。
  5. 从小做起，按需演进。从简单的查询路由开始，最多两周就能落地。随着数据源增多，逐步引入 Function Calling、联邦检索。
